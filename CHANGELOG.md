## shipyard

### 1.8.35 — 2026-06-01

Closes [#452](https://github.com/mattsears18/shipyard/issues/452) (P1, `bug`) via [#454](https://github.com/mattsears18/shipyard/pull/454) — **makes the primary-checkout leak guard derive `PRIMARY_CHECKOUT` independent of the orchestrator's Bash cwd, so a harness-leaked `agent-*` cwd can't make the guard mutate the wrong worktree.** The Claude Code `isolation: "worktree"` dispatch path can silently relocate the orchestrator's own Bash-tool cwd into a just-returned agent's worktree (same harness-env leak class as #387 / #322 / #354). The A.0.6 primary-leak guard derived `PRIMARY_CHECKOUT` from `git rev-parse --show-toplevel` with an `orchestrator-*`-only suffix strip, so when cwd had leaked into an `agent-*` worktree the strip didn't match — the guard read the agent worktree's branch (`do-work/issue-NNN`), saw it `!= main`, and ran `git checkout main && git pull --ff-only` against the **agent** worktree while emitting a phantom `[primary-leak] restored primary` line, never inspecting the real primary. The fix derives the primary cwd-independently from `git worktree list --porcelain`'s first entry (always the primary), retaining the suffix-strip — now covering `agent-*` as well as `orchestrator-*` — as a fallback.

- **`plugins/shipyard/commands/do-work/steady-state.md`** — A.0.6 primary-leak guard now derives `PRIMARY_CHECKOUT` from `git worktree list --porcelain`'s first entry; suffix-strip fallback extended to `agent-*`.
- **`plugins/shipyard/commands/do-work/drain.md`** — drain-entry primary-leak guard given the identical cwd-independent derivation.
- **`plugins/shipyard/scripts/tests/do-work-split.test.sh`** — 6-assertion regression block asserting the primary derivation is correct when invoked from inside an `agent-*` worktree (149 tests green, shellcheck clean).

### 1.8.34 — 2026-05-31

Closes [#434](https://github.com/mattsears18/shipyard/issues/434) (P1, `bug`) via [#451](https://github.com/mattsears18/shipyard/pull/451) — **adds a hard verify-before-file gate so audit agents can't file findings against unverified command output.** During a full `/shipyard:audit all` run against `mattsears18.com`, 13 of 44 filed issues were fabricated false positives that had to be retracted — four auditors ran reconnaissance reads in the *same* parallel tool-call batch as `gh issue create` and acted on unverified (sometimes empty/garbled) output (e.g. "`tailwind.config.ts` doesn't exist" against a repo where it does; "remove unused `@tailwindcss/typography`" against a package that isn't a dependency), then self-corrected and retracted *after* filing. The skills didn't hard-gate filing behind a verification step. The fix is structural and three-sided: the filing skill now requires every finding to cite a freshly-read evidence artifact captured in a step that *completed before* the create (never batched with it), plus a self-review re-read pass immediately before each create and a documented false-positive retraction path; the rubrics Evidence bar requires the evidence be freshly observed (not assumed) and observed before the create; and `/shipyard:audit`'s agent prompt carries the verify-before-file instruction while its reconciliation step closes any issue an agent flags as filed-in-error.

- **`plugins/shipyard/skills/filing-github-issues/SKILL.md`** — new "Verify before you file (REQUIRED — hard gate)" section (sequence recon strictly before the create, confirm each claim against ground truth, self-review re-read pass, drop inconclusive findings) plus the "If you filed a false positive anyway: the retraction path" subsection (self-close with a retraction comment, or surface under a "Filed in error (needs retraction)" summary line when the close is classifier-denied).
- **`plugins/shipyard/skills/audit-rubrics/SKILL.md`** — Evidence bar now requires evidence freshly observed this session (not assumed) and observed *before* the create, cross-referencing the filing-skill gate.
- **`plugins/shipyard/commands/audit.md`** — agent prompt template now carries the verify-before-file instruction and asks agents to surface filed-in-error issues; reconciliation step gains a "close any issue an agent flagged as filed in error" rule.
- **`plugins/shipyard/scripts/tests/audit-verify-before-file.test.sh`** (new) — 17 regression assertions guarding the verify-before-file gate, the before-create sequencing, the self-review pass, the retraction path + in-summary fallback, the rubrics cross-reference, and the audit.md prompt + reconciliation wiring.

### 1.8.33 — 2026-05-31

Closes [#435](https://github.com/mattsears18/shipyard/issues/435) (P1, `bug`) via [#TBD](https://github.com/mattsears18/shipyard/pull/TBD) — **stops parallel auditors from reporting guessed/colliding issue numbers and gives the orchestrator a trustworthy agent→issue mapping to reconcile against.** When `/shipyard:audit all` dispatches 12+ auditors at once, GitHub assigns issue numbers sequentially across the concurrent `gh issue create` calls, so an agent that predicts its number or reads it back with a separate post-filing `gh issue list` can report a colliding or stale value — and a genuine finding can be silently lost (a 15-auditor `mattsears18.com` run had two auditors both report #150/#152 and one real finding never traced to its actual number). - **`plugins/shipyard/skills/filing-github-issues/SKILL.md`** — new "Capture the real issue number — never guess or read it back stale" section (captures `gh issue create` stdout into a variable), "Per-run attribution marker" section (the conditional `audit-run=<run-id>` body comment, volatile and NOT used for dedup), and "Return summary — report captured URLs" section; body template now shows the conditional `audit-run` comment alongside the mandatory `audit-key`.
- **`plugins/shipyard/commands/audit.md`** — new "Generate the audit run id" pre-dispatch step; agent prompt template now passes the run id and the capture-from-stdout instruction; new "Reconcile filed issues before summarizing" step cross-checks reported URLs against a `gh issue list --search "audit-run=<id>"` enumeration; end-of-run summary now lists captured URLs and a Process-notes mismatch surface.
- **`plugins/shipyard/scripts/tests/audit-issue-number-reconciliation.test.sh`** (new) — 18 regression assertions guarding the capture-from-stdout requirement, the predicted/read-back prohibitions, the `audit-run` marker + its not-for-dedup warning, and the orchestrator's run-id generation, prompt injection, and reconciliation step.

Closes [#440](https://github.com/mattsears18/shipyard/issues/440) (P2, `shipyard`) via [#447](https://github.com/mattsears18/shipyard/pull/447) — **adds worker-preamble guidance for when GitHub push-protection blocks a synthetic test-fixture secret.** A worker that adds a NEW test fixture with a realistic-shaped secret (Slack `xoxb-`, Stripe `sk_live_`/`sk_test_`, GitHub `ghp_`/`github_pat_`, AWS `AKIA…`) — to exercise a scrubber, secret-scan rule, or redaction regex — gets its `git push` bounced by GitHub's server-side push-protection, even when the fixture is already allowlisted in `.gitleaks.toml`. Bails with `blocked:` for the rare checksum-validating-detector case.

### 1.8.30 — 2026-05-31

Closes [#404](https://github.com/mattsears18/shipyard/issues/404) (P1, `audit:testing`) — **ships a branch-protection enablement helper + contributor docs so the maintainer can make the CI gates *required* with one reviewable command.** The repo's `tests.yml` / `shellcheck.yml` / `secret-scan.yml` workflows run on every PR but `main` has no branch protection, so they're advisory only — a red-tests PR can still merge (9 of the last 100 `tests.yml` runs on `main` were failures). Required contexts: `bash test suites`, `shellcheck`, `shell tests`, `gitleaks`. `REPO` / `BRANCH` env overrides for reuse.
- **`plugins/shipyard/scripts/tests/enable-branch-protection-script.test.sh`** (new) — 13 assertions: existence / executability / shebang, every required-context name is referenced, `--help` / unknown-flag behavior, the default mode is a non-mutating dry-run (asserts the apply path does NOT run, pointed at a non-existent repo so it can't touch live settings), and shellcheck-clean.

### 1.8.28 — 2026-05-31

Closes [#436](https://github.com/mattsears18/shipyard/issues/436) (P1, `shipyard`) — **conflict markers can no longer ride a green CI run onto `main`**. During a high-concurrency `/shipyard:do-work` session a fix-rebase/merge force-pushed a `CHANGELOG.md` carrying unresolved Git conflict markers (`=======` / `>>>>>>> <sha>`) which then merged to the default branch; none of the existing gates (shellcheck, bash tests, secret-scan) grep for markers, so the corruption merged silently and every subsequent PR branched from a poisoned base (repro: session `do-work-20260531T172554Z-8676`). The fix is two-layer: a repo-side CI gate that fails on any conflict marker in a tracked file, plus a worker-side assertion in the fix-rebase spec that bails `blocked rebase` before force-pushing if a marker survived conflict resolution. The detection regex is the anchored exactly-seven-chars-then-space-or-EOL pattern shared by the CI gate, the worker assertion, and the existing `check-merge-conflict` pre-commit hook, so all three agree on what counts as a marker and benign inline prose (a doc writing `` `=======` `` in backticks) doesn't false-positive. Patch bump (new CI gate + worker-spec assertion; no behavior change to non-rebase paths).

- **`plugins/shipyard/scripts/conflict-marker-scan.sh`** — new reusable scanner: `git grep`s tracked files for the anchored marker pattern, prints offending `file:line`, exits 1 on any match. Honors a `conflict-marker-scan: allow` opt-out directive for fixtures that legitimately construct conflicted files.
- **`.github/workflows/conflict-markers.yml`** — new read-only CI gate running the scanner on every push to `main` + PR; separate-concern / separate-file convention matching the other gates, checkout action pinned by SHA, no event-payload interpolation.
- **`plugins/shipyard/agents/issue-worker/fix-rebase.md`** — new step 5.5: assert no conflict markers remain (same regex) before the force-push, bail `blocked rebase #<M>: conflict markers remain after resolution` if any survive — the worker-side half of the defense so a fix-rebase dispatch isn't the thing that needs catching.
- **`plugins/shipyard/scripts/tests/conflict-marker-scan.test.sh`** — 13 regression assertions: artifact presence, workflow wiring + least-privilege, fix-rebase.md assertion + bail string, and behavioral scanner checks (clean pass / marker fail / allow-directive opt-out / no false-positive on wrong-length or inline marker-like prose).

### 1.8.27 — 2026-05-31

Closes [#437](https://github.com/mattsears18/shipyard/issues/437) (P1, `shipyard`) — **batch dispatch now pre-allocates distinct manifest versions via a session-local `version_cursor`, killing the C≥2 pool-fill version-row collision storm**. The #339 next-available-version computation walks `session_prs` for **open** PRs to find the highest manifest version any in-flight PR has claimed. At the initial pool fill (setup step 7) and any step C multi-fill, the N sibling workers' PRs are not open yet, so every worker in the batch reads the same floor (main's version) and the per-dispatch walk hands them all the same next slot — the first C=4 batch all claimed 1.8.2, and 3 of 4 went DIRTY on the version row, eating a drain-rebase storm (repro: session `do-work-20260531T172554Z-8676`). The fix adds a session-local `version_cursor` high-water mark tracking the highest version slot **claimed by dispatch** (not just by open PRs): the computation seeds from `max(version_cursor, session_prs-walk)` and **advances the cursor to the value it hands out**, so running the block once per slot while composing a batch's prompts yields distinct monotonic slots (`main+1`, `main+2`, … `main+N`). The `Agent` calls still fire simultaneously; only the per-slot version assignment is computed serially against the shared cursor. Session-local working memory (not mirrored to the session-state file), consulted only when `version_coordination.enabled` + a `manifest_path` is configured — a no-op on non-coordinated repos. Patch bump (orchestrator dispatch-spec behavior, default-safe on non-coordinated repos).

- **`plugins/shipyard/commands/do-work.md`** — new `version_cursor` orchestrator-state struct (15th entry; opening-sentence count word advanced thirteen → fifteen) documenting the high-water-mark semantics, the batch-blindness gap it closes, and its session-local lifecycle.
- **`plugins/shipyard/commands/do-work/steady-state.md`** — the #339 next-available-version computation now folds `version_cursor` into `max_inflight_version` and advances the cursor to each handed-out value; a new paragraph explains why the cursor (not the `session_prs` walk alone) is what makes a *batch* of simultaneous dispatches monotonic.
- **`plugins/shipyard/commands/do-work/setup.md`** — step 7 (initial pool fill) documents running the version computation once per issue-work slot in sequence, so each batch sibling's prompt carries a distinct "Next-available version" paragraph.
- **`plugins/shipyard/scripts/tests/do-work-split.test.sh`** — six `#437` regression assertions (struct-list naming + issue citation in do-work.md, steady-state seed/advance, setup batch pre-allocation) plus the updated "fifteen mental data structures" count-word assertion.

### 1.8.26 — 2026-05-31

Closes [#438](https://github.com/mattsears18/shipyard/issues/438) (P1, `shipyard`) — **serializes drain-phase rebases on CHANGELOG-coordinated repos and warns when `allow_auto_merge=false` + admin silently direct-merges**. On a repo where every PR appends a top-of-file `### <version>` CHANGELOG entry (the canonical version-coordinated shape — `version_coordination.enabled` + a non-empty `changelog_path`, the shipyard plugin itself), parallel drain rebases cannot converge: each merge moves the CHANGELOG insert point, re-DIRTYing every sibling that just rebased onto the previous head. The #438 repro saw 6 distinctly-versioned PRs (1.8.18–1.8.23) all re-DIRTY on the CHANGELOG append after the first merge; only serial rebase (rebase one → let it merge → rebase the next) converged. A new `version_coordination.serialize_drain_rebase` key (default `true`) caps effective drain-rebase concurrency to 1 when the gate is engaged — one DIRTY PR rebased per poll, the next deferred until the in-flight one merges. The cap applies only to fix-rebase dispatch; fix-checks against red PRs stays urgent. Separately, a setup-time warning fires when the repo has `allow_auto_merge=false` and the dispatching user has admin (the silent-direct-merge shape from #340), explaining why C≥2 version coordination can't hold without serialized merges. Patch bump (additive config key + drain-spec behavior, default-safe on non-coordinated repos).

- **`plugins/shipyard/schemas/shipyard.config.schema.json`** — registers `version_coordination.serialize_drain_rebase` (boolean, default `true`) with the convergence rationale and the engage condition (enabled + non-empty `changelog_path`).
- **`plugins/shipyard/scripts/shipyard-config.sh`** — adds `serialize_drain_rebase: true` to the built-in `version_coordination` defaults so `get` never trips a key-not-present fallback.
- **`plugins/shipyard/commands/do-work/drain.md`** — new CHANGELOG-serialization gate in per-poll action 2 (after the #323 CI-minute truncation): when engaged, caps the `D_dirty` dispatch set to one PR per poll and defers the rest, with a no-op path for non-coordinated repos.
- **`plugins/shipyard/commands/do-work/setup.md`** — new step 1.3 detects the `allow_auto_merge=false` + admin shape via a single REST read folded into the setup parallelization batch and emits a one-line operator warning (no behavior change — the orchestrator never flips repo auto-merge config).

### 1.6.0 — 2026-05-29

Closes [#378](https://github.com/mattsears18/shipyard/issues/378) (P1, feat) — **phase 1 of the cross-PR flake registry**. Today's `fix-checks-only` worker handles flakes per-PR in isolation: none of the independent workers sees that the SAME test has flaked on N other PRs this week, so chronic flakes go uninvestigated and the user's repo-level rule ("Flaky CI is a real failure — fix the root cause") is silently violated. This phase ships the **data layer + threshold tooling**: a session-spanning `~/.shipyard/flake-registry.jsonl` ledger, a `fix-checks-only` write path that records one event per flake conclusion (gated on `flake_registry.enabled`, default off), the `shipyard.config.json::flake_registry` config schema (window / rerun-threshold / distinct-PRs-threshold / actions), and a `flake-registry.sh crossed` reader that aggregates per-(workflow,job,test) flake rates over the window and names which flakes have crossed the escalation bar. Per issue-work scope discipline the *enforcement* of the three escalation actions (`file-tracking-issue`, `stop-auto-rerunning`, `apply-blocked-ci`) is deliberately deferred to a follow-up phase ([#385](https://github.com/mattsears18/shipyard/issues/385)) — `crossed` emits the configured action set on each crossed row so the future enforcement phase can act on it, but this phase files no issues, writes no flake-suspects file, and labels no PRs. Minor bump (new feature, additive — defaults preserve pre-#378 behavior).

### 1.5.29 — 2026-05-29

Closes [#374](https://github.com/mattsears18/shipyard/issues/374) (P2, bug) — the **end-of-session drain's 120-min wall-clock ceiling exited prematurely on healthy-but-slow merge trains**. The old ceiling fired on the wrong signal: a multi-PR session with full E2E sharding (3 shards × ~10-15 min), Lighthouse (~9 min), and release-please cascades (each merge re-DIRTY-ing siblings, then a 15-25 min rebase + re-CI round-trip) can legitimately take >2h to land even when nothing is wrong, and the cap cut those sessions off mid-flight — forcing the user to re-invoke `/do-work` to resume the drain (which also resets cost tracking). The fix replaces wall-clock with a **per-PR progress signal**: drain tracks each open PR's `headRefOid` and the timestamp it last moved (`head_unchanged_since`), and a PR counts as "settled" once its head has been unchanged for `ci.settled_minutes` (default 20) — CI grinding but the branch not being actively reworked. Drain exits the moment every session_pr is settled (merged/closed, `blocked:ci`, rebase-blocked, rate-limit-capped, or pending-head-quiet), with no fixed-poll no-progress window. A new `ci.max_drain_hours` (default 8h) survives as the belt-and-braces outer bound so a degenerate session (runaway CI loop, infinite rebase whose head moves every poll) still can't run forever. The per-poll status line now surfaces the progress clock per still-pending PR (`PR #N: head unchanged 12m / 20m settled-threshold`) so the operator can watch the signal that determines settle. Docs/spec + config-schema + test change — the drain loop is orchestrator-interpreted prose, so the behavior change lands in `drain.md` rather than a helper script. Companion to #373 (cross-session DIRTY adoption into `session_prs`, shipped earlier this session).
