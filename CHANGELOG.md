## shipyard

### 1.9.30 — 2026-06-14

**Drain no longer dispatches fix-rebase on DIRTY-AND-red PRs — the per-poll bookkeeping now splits DIRTY PRs by rollup status and routes DIRTY+failing to fix-checks directly** (#577, PR #580). The bug: when a PR ended the session with `mergeStateStatus == DIRTY` AND a hard-failing check (e.g., a unit-test FAILURE introduced by a sibling merge), the drain's `D_dirty` classifier excluded it correctly (the spec already said "no hard-failure check"), but the orchestrator — reading only `mergeStateStatus` — dispatched fix-rebase anyway. The fix-rebase worker immediately bailed with `blocked rebase: PR has failing checks — needs fix-checks, not rebase`, wasting a Haiku dispatch plus an extra round-trip before the orchestrator finally re-routed to fix-checks. The fix introduces an explicit `D_dirty_red` set (DIRTY + hard-failure rollup) that is routed to fix-checks in per-poll action 1 alongside `R_new`, and updates the `D_dirty` definition to add a "never classify from `mergeStateStatus` alone" prohibition and an explicit `AND NOT in D_dirty_red` guard. The drain status line gains a `dirty_red=<count>` token (omitted when 0) so the routing decision is visible in drain logs.

### 1.9.29 — 2026-06-14

**Drain fix-checks/fix-rebase workers no longer bail "branch locked in another worktree" — completed issue-work agent worktrees are now force-reaped on `peer-alive` locks** (#576, PR #577). During the end-of-session drain, dispatching `fix-checks` or `fix-rebase` workers against PRs opened this session repeatedly failed because the original issue-work agent's worktree was still checked out on the `do-work/issue-<N>` branch. The A.1 `shipped` immediate-reap path and the drain pre-dispatch head-branch reap both deferred conservatively on `peer-alive` lock classifications — but a completed issue-work agent is logically done by definition (its terminal return string is in hand), so deferring caused the head branch to remain locked throughout drain, forcing every fix-checks and fix-rebase worker to bail. The fix applies the same force-reap-on-peer-alive posture A.0.5 uses for crash returns: (1) the A.1 `shipped` immediate-reap path now force-reaps even on `peer-alive`, auditing the override with classification `peer-alive-force`; (2) the drain pre-dispatch reap now distinguishes completed issue-work worktrees (branch pattern `do-work/issue-*`) from live fix-checks/fix-rebase workers (which must not be yanked) — force-reaped with classification `peer-alive-force-drain`, while other branch patterns still preserve the conservative defer. Both changes carry the `~/.shipyard/reap-audit.jsonl` classification tokens so an operator can trace the override. A regression test guards that neither path regresses to a peer-alive defer for completed issue-work worktrees.

- **`plugins/shipyard/commands/do-work/steady-state.md`** — A.1 `shipped` immediate-reap block: `peer-alive` defer branch replaced with force-reap; prose extended with "Force-reap even on `peer-alive` here" rationale naming #576; closing "fire-and-forget" paragraph updated (peer-alive defers are no longer the fall-through path for the A.1 shipped site).
- **`plugins/shipyard/commands/do-work/drain.md`** — pre-dispatch head-branch reap: `peer-alive` defer replaced with a `case` branch that force-reaps for `do-work/issue-*` worktrees and defers for all other branch patterns; prose extended with "Force-reap even on `peer-alive` at the drain dispatch site" rationale; "Why it's safe to reap" section extended with #576 context; failure-mode prose updated to reference the #576 repro; closing commentary updated to name both classification values.

### 1.9.28 — 2026-06-14

**A.0.5 crash-recovery now completes the version-coordination bump (manifest + CHANGELOG) before pushing the recovered PR** (#575, PR #576). The previous crash-recovery path (A.0.5 in `steady-state.md`) salvaged committed or dirty-worktree work by auto-committing and pushing a recovery PR — but did not add the manifest version bump or CHANGELOG entry the crashed worker had not yet reached. On version-coordinated repos (like `shipyard` itself) where the release rule mandates a bump per merged PR and admin + no-required-checks makes the direct-merge path immediate, a crash-recovered PR would auto-merge with no version bump, shipping the change invisibly to all installations. The fix adds a `a05_version_bump` helper that fires in both recovery branches (committed-but-unpushed and dirty-worktree) before the push: it reads `version_coordination.enabled`, compares the worktree manifest version to `origin/<default>`, and when they match (the worker never reached its own bump step), computes the next available version as a patch bump, rewrites the manifest file, and prepends a minimal CHANGELOG stub referencing the recovered issue. For the dirty-worktree path the bump folds into the auto-commit; for the committed-but-unpushed path it lands as a dedicated bump commit on top. The bump is fire-and-forget: when it can't be computed (manifest read fails, jq absent, coordination disabled), the recovery logs a loud `[reconcile-A.0.5-recovery] WARNING: recovered PR has no version bump — manual release bump required` advisory and pushes the PR anyway. Non-version-coordinated repos are entirely unaffected.

- **`plugins/shipyard/commands/do-work/steady-state.md`** — A.0.5 "Pre-reap recovery check" section extended with a "Version-coordination bump during recovery" prose block and the `a05_version_bump` bash helper function; the function is called in both the committed-but-unpushed branch (step 1.5, adds a bump commit before the push) and the dirty-worktree branch (step 1.5, applies file edits before `git add -A` so the bump folds into the auto-commit); the WARNING advisory log line for the can't-compute case is specified; non-coordinated repos explicitly documented as unaffected; issue #575 referenced inline.

### 1.9.27 — 2026-06-14

**The `shipyard` provenance label is now stamped on EVERY issue and PR shipyard creates, and every creation path ensures the label exists first** (#573, PR #TBD). Previously the stamp was applied inconsistently: `/do-work` stamped its PRs, but `/shipyard:file-issue` explicitly *declined* to stamp issues (its "Don't apply the `shipyard` label" rule), and no path guaranteed the label existed before filing — so on a repo where the label was never bootstrapped (a consumer repo that skipped `/shipyard:init`, or where the best-effort background `gh label create` raced/failed), shipyard-authored artifacts came out unlabeled and untraceable. The fix makes the rule uniform: every creation path applies `--label shipyard` and first runs an idempotent `gh label create shipyard --description "Worked on by /shipyard:do-work" --color 5319E7 2>/dev/null || true` so a missing label is created rather than silently dropped. The previously-conflicting conventions are reconciled — `/shipyard:file-issue`'s "Don't apply the `shipyard` label" rule is flipped to an affirmative stamp, and `CLAUDE.md`'s session-stamp section is broadened from "PRs only" to "every issue AND PR." The `audit:<dimension>` origin label and the `shipyard` session-stamp stay orthogonal: auditor-filed issues carry both.

### 1.9.26 — 2026-06-14

**Scope-preflight no longer re-gates issues a human has already cleared — honors label-timeline gate-clears and `<!-- do-work-decision-resolved -->` sentinel comments** (#569, PR #TBD). A new `<!-- do-work-decision-resolved -->` sentinel convention is documented in `CLAUDE.md` so `/my-turn`-driven walkthroughs and human maintainers know to stamp it when recording decisions.

### 1.9.25 — 2026-06-13

**New shared `shipyard:auditing-authenticated-surfaces` skill — the safe, framework-agnostic pattern for live-URL auditors that tour login-walled surfaces, plus wiring from the `web-ux` and `a11y` auditors** (#567, PR #TBD). The live-URL auditors previously carried only a thin "optionally: test credentials for authenticated surfaces" line and told the agent to type the email/password into the browser MCP itself — an unsafe model (the secret lands in transcripts/tool logs) and an unreliable one (a fresh signup can't clear email-verification / OAuth / age gates; a saved-then-reloaded SPA session comes back logged-out). The reusable kernel for doing it correctly lived only in a consumer repo's `CLAUDE.md`, so every login-walled audit re-derived it. This lifts that kernel into a shared skill (sibling to `audit-rubrics` / `filing-github-issues`) capturing four rules: (1) auditors must NOT self-authenticate by typing secrets — use a login harness that reads creds from a gitignored env file, never echoes them, and captures artifacts (screenshots / axe-core JSON / DOM probes) for the auditor to judge; (2) require a pre-provisioned account; (3) the SPA `storageState` gotcha (Firebase Web SDK et al. store the token in IndexedDB/localStorage, which Playwright `storageState` does NOT serialize — log in and tour within ONE live context); (4) assert on a protected route that 302s when unauthenticated, not on `/` (which renders the public marketing view when logged out). Project-specific wiring (which account, which env file, which seed command) stays in the consumer repo.

### 1.9.23 — 2026-06-13

**`/my-turn` ranks within each tier by a leverage score, not raw age — the single `→ Next:` directive no longer surfaces a stale undecomposable epic** (#565, PR #TBD). The within-tier secondary sort was a flat `createdAt` ascending (oldest-first), which in single-action mode made the *stalest* item the sole `→ Next:` directive. On a P0 tier dominated by long-lived `needs-human-review` issues, oldest-first regularly floated an auto-undecomposable epic — the *least* actionable item — to the top, directly contradicting the command's "single highest-leverage thing blocked on you" promise (repro: a 4-week-stale `epic(aso)` #540 ranked #1 over genuinely high-leverage one-decision issues). The fix replaces the flat oldest-first secondary sort with a **leverage score** (derived entirely from signals passes A–D already collect — no new `gh` calls): pure-decision items that flip to dispatch-eligible with one human call score highest; quick external/console actions next; on-device/multi-party verification below that; auto-undecomposable epics (`<!-- do-work-needs-decomposition -->` + `couldn't auto-decompose:` sentinel) score lowest and sink. `createdAt` ascending remains the final tie-breaker within a leverage score.

### 1.9.22 — 2026-06-12

**Scope-result freshness window eliminates redundant scope-agent re-dispatches when fresh marker-tagged diagnosis comments already exist on issues** (#563, PR #564). Previously, the orchestrator re-dispatched scope agents every session even when prior sessions had already documented their conclusions as marker-tagged comments (`<!-- do-work-external-dependency -->`, `<!-- do-work-human-decision-required -->`, `<!-- do-work-needs-decomposition -->`) on the issues. The repro: a maintainer bulk-cleared `needs-human-review` labels from 19 open issues; the orchestrator launched 7 scope agents (~330k tokens) that returned nearly verbatim re-derivations of 1–4 day-old comments. The fix introduces `scope.diagnosis_reuse_hours` (config knob, default 72h): before dispatching a scope agent, the orchestrator reads the issue's newest marker-tagged diagnosis comment; if it is within the window AND its mechanical evidence re-validates cheaply (cited blockers still open, body unchanged since the comment), it records the defer with provenance `cached-diagnosis` and skips the agent dispatch. Drain.md 5.b is extended to cover two tiers: the existing same-session carve-out (Tier 1, #549) and a new cross-session tier (Tier 2, #563) that re-validates `cached-diagnosis` entries within the configured window before dispatching a fresh scope agent.

### 1.9.20 — 2026-06-12

**Scoping agents now MUST include shared regression-test suite files in `files` for fix-class issues, preventing drain-phase rebase bails caused by unclaimed same-file edits** (#554, PR #561). When two sibling PRs edit the same shared test accumulator file (`do-work-split.test.sh` in this repo), the one whose scope agent omitted the file from `claimed_paths` causes the orchestrator's hard-collision check to miss the overlap. Mirrors the coordination-managed-paths contract (`plugin.json` + `CHANGELOG.md`) that was already implicit in worker behavior but not agent-prompt-visible. Config-driven `always_claimed_paths` surface deferred to phase 2 (requires schema + orchestrator coordination changes).

### 1.9.19 — 2026-06-12

**New `changelog-monotonicity-scan.sh` CI gate + worker-side assertions close the silent released-entry deletion gap** (#555, PR #559). Previously a resolved-wrong merge conflict could silently drop released `### <version>` headings from `CHANGELOG.md` with no CI gate firing — the conflict-marker gate (#436) only catches *unresolved* markers. Session `do-work-20260612T115814Z-28614` demonstrated the failure: PR #552 and #553 resolved their CHANGELOG conflicts correctly (no markers survived) yet both dropped `### 1.9.10` and the `### 1.9.9` heading from `main`; the loss was only noticed when a human eyeballed the file during a later manual rebase of PR #551. The fix is two-layer, mirroring #436's approach: (1) a new `changelog-monotonicity-scan.sh` script that diffs the `### <version>` heading set between the merge-base and the PR head, failing when any base-side heading is absent on the head; and (2) worker-side assertions in `fix-rebase.md` (step 5.6) and `issue-work.md` (release step) that run the scan before committing or force-pushing and bail with `blocked` if any released heading was deleted. The scan is false-positive-safe: only headings present on the merge-base count, so a PR re-numbering its own unmerged entry (which was never on the base) does not trip it. A new `changelog-monotonicity-scan.test.sh` test suite unit-tests the scanner (clean pass, deletion detected, opt-out directive, false-positive guard, multi-deletion, brand-new CHANGELOG) and also runs the scanner against the working-tree CHANGELOG to pin the live-repo invariant, mirroring the #436 test pattern.

### 1.5.29 — 2026-05-29

Closes [#374](https://github.com/mattsears18/shipyard/issues/374) (P2, bug) — the **end-of-session drain's 120-min wall-clock ceiling exited prematurely on healthy-but-slow merge trains**. The old ceiling fired on the wrong signal: a multi-PR session with full E2E sharding (3 shards × ~10-15 min), Lighthouse (~9 min), and release-please cascades (each merge re-DIRTY-ing siblings, then a 15-25 min rebase + re-CI round-trip) can legitimately take >2h to land even when nothing is wrong, and the cap cut those sessions off mid-flight — forcing the user to re-invoke `/do-work` to resume the drain (which also resets cost tracking). The fix replaces wall-clock with a **per-PR progress signal**: drain tracks each open PR's `headRefOid` and the timestamp it last moved (`head_unchanged_since`), and a PR counts as "settled" once its head has been unchanged for `ci.settled_minutes` (default 20) — CI grinding but the branch not being actively reworked. Drain exits the moment every session_pr is settled (merged/closed, `blocked:ci`, rebase-blocked, rate-limit-capped, or pending-head-quiet), with no fixed-poll no-progress window. A new `ci.max_drain_hours` (default 8h) survives as the belt-and-braces outer bound so a degenerate session (runaway CI loop, infinite rebase whose head moves every poll) still can't run forever. The per-poll status line now surfaces the progress clock per still-pending PR (`PR #N: head unchanged 12m / 20m settled-threshold`) so the operator can watch the signal that determines settle. Docs/spec + config-schema + test change — the drain loop is orchestrator-interpreted prose, so the behavior change lands in `drain.md` rather than a helper script. Companion to #373 (cross-session DIRTY adoption into `session_prs`, shipped earlier this session).
