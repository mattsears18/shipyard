### 2.3.5 — 2026-07-11

**Harden the `CLAUDE_PLUGIN_ROOT` re-export preamble so it can't resolve to a stale `.bak` marketplace copy** (closes #681; PR #TBD). The step-0.3 idempotent fallback that every `${CLAUDE_PLUGIN_ROOT}/scripts/*.sh` call routes through globbed `marketplaces/*/plugins/shipyard | head -1` when the harness leaves the env var empty. Because `ls -d` sorts and `.` (`0x2E`) sorts before `/` (`0x2F`), a sibling `shipyard.bak` (or `.old`/`.orig`/`.disabled`) directory won `head -1` over the real `shipyard` dir — silently running helper scripts from a backup copy (observed on `mattsears18/lightwork`, session `session_01Hs4CqGT53F6kwVasHiyLnH`). The glob also targeted `marketplaces/` while the loaded install lives under `cache/<mp>/<plugin>/<version>/`, so the two could diverge in version even with no `.bak` present. The rewritten fallback probes in a safer order: repo-local dogfood → **authoritative `installed_plugins.json` `installPath`** (the exact loaded install, immune to both failure modes) → **hardened marketplace glob** (tries the exact `marketplaces/shipyard` path first, excludes `.bak`/`.old`/`.orig`/`.disabled` siblings, requires a real `scripts/` dir) → repo-local error path. Step 0.4 now also echoes the resolved value once (`resolved CLAUDE_PLUGIN_ROOT=… >&2`) so a wrong pick is visible in the session log instead of resolving invisibly. Spec/docs + test only; no helper-script runtime change. Files touched:

- `plugins/shipyard/commands/do-work/setup/00-config-worktree.md` — rewrote the canonical step-0.3 preamble and its semantics prose (now a 4-layer probe), and added the one-shot `echo "resolved CLAUDE_PLUGIN_ROOT=…" >&2` to the step-0.4 block.

### 2.3.2 — 2026-07-06

**`fix-rebase.md` §4.6 version-leapfrog resolution is now bump-level-aware — a rebased major/minor PR no longer gets downgraded to a patch release** (#673, PR #TBD). §4.6's version-coordinated conflict-resolution recipe (the #466 carve-out) re-numbered a version-leapfrogged PR to *main's floor + next free **patch***, the same patch-only assumption #671 (PR #672) just fixed in the dispatch-time next-available-version computation. The trap: a PR pre-allocated a **major** (breaking) or **minor** (feature) release that then gets version-leapfrogged during a drain-phase rebase was silently collapsed to a patch — e.g. a PR pre-allocated `3.0.0` whose main advanced to `2.4.5` resolved to `2.4.6`, shipping the breaking change as a patch. §4.6 step 1 now infers the PR's intended release level from the **ground truth the PR already encodes** — the delta between its own pre-allocated manifest version and its merge-base with main (read from `origin/$HEAD_REF`, a stable ref independent of the in-progress rebase) — rather than a patch assumption or the issue title, then bumps main's floor at *that* level using the same major → `(X+1).0.0` / minor → `X.(Y+1).0` / patch → `X.Y.(Z+1)` shape #671 landed. The floor stays the hard collision constraint; only the semver level is level-aware, so the benign over-estimate edge case (a patch PR pre-allocated above an in-flight minor sibling's floor) bumps one level high but never below the floor and never collides — strictly safer than the downgrade this fixes. ### 2.3.1 — 2026-07-06

**Version-coordination next-available-version is now bump-type-aware, no longer patch-only** (#671, PR #TBD). The per-dispatch next-available-version computation in `dispatch-rules.md` (and the parallel A.0.5 crash-recovery bump in `steady-state.md`, #575) only ever computed a patch bump (`max_inflight_version + 1` on the Z segment), blind to issues that explicitly require a major (breaking) or minor (feature) release. On a version-coordinated repo that cuts a release per PR, the injected "Next-available version (orchestrator-supplied)" paragraph then contradicted the issue body's stated version intent with a semver-wrong "use this exact value" directive — the latent trap the #659 epic's MAJOR/minor shards would have hit (main `1.27.1`: #661 needed `2.0.0` but the formula gave `1.27.2`; the feature shards needed minor bumps but got patch). The computation now infers the bump level from the dispatched issue's Conventional Commits title/body and computes the next free slot at that level; the `version_cursor` still advances to the exact computed value so a batch of N dispatches stays strictly monotonic (collision-avoidance guarantee preserved), and the injected paragraph is weakened from a patch-only "use this exact value" to a floor + recommended-slot form that leaves the semver level to the worker (the issue body stays the single source of truth for release intent). ### 2.2.1 — 2026-07-06

**Fast worktree-lock reap: rename + prune + backgrounded delete so a large `node_modules/` can't hang the reap** (#664, phase d of #659, PR #TBD). `git worktree remove --force` recursively unlinks the worktree directory *inline*, which stalls for many seconds to minutes when the worktree carries a large `node_modules/` (tens of thousands of small files) — and the branch checked out there stays locked until the remove completes, blocking `fix-checks` retries that need the head branch ("head branch locked in another worktree"). The new fast path decouples branch-freeing from bulk deletion: `git worktree unlock` → `mv <wt> <wt>.reap-dead-<pid>-<epoch>` (instant inode relink on the same filesystem) → `git worktree prune` (drops the now-missing registration, freeing the branch immediately) → backgrounded `rm -rf <dead> &` (the only expensive step, detached so it can't stall the caller). Steps up to the prune are synchronous and independent of tree size; only the recursive unlink is backgrounded, and the path falls back to the synchronous `git worktree remove --force` when the rename can't be performed (path missing, cross-device mount, permission, race). Both reap actions in the `reap` helper — `reaped` (agent worktrees) and `reaped-orphan-orchestrator` (setup 1.6.5 crash-orphan reap) — route through it, so every reap call site that funnels through the #284 single-source-of-truth helper (setup 3b, steady-state pre-dispatch reap, drain pre-dispatch reap, cleanup-summary) gets the fast path with no per-site change. Files touched:

- `plugins/shipyard/scripts/worktree-reap.sh` — adds the `fast_worktree_remove` helper (unlock + rename-aside + prune + backgrounded delete, with fallback), routes the `reaped` and `reaped-orphan-orchestrator` reap actions through it, and updates the `reap`-subcommand docstring; the crash-orphan case now takes the fast path, so the `-raw-rm` / `-failed` audit variants are reserved for the genuine rename-then-remove last resort.
- `plugins/shipyard/scripts/tests/worktree-reap.test.sh` — updates the orphan-orchestrator crash-orphan test (34/34a/34b) to the new fast-path action, and adds a fast-reap regression group (76–77) asserting the rename → prune → branch-freed sequence on a real registered worktree, audit-line continuity, and the already-removed-path no-op fallback.

### 2.1.0 — 2026-07-06

**Redefine the `/do-work` termination contract: complete only when every workable issue is closed/dispositioned AND every session PR is merged (or confirmed external-blocked) — not when the dispatch pool drains; drive the drain tail toward that** (#662, phase b of #659, PR #TBD). Previously "backlog empty + in-flight drained" read as the finish line: the dispatch loop stopped, the end-of-session drain watched the merge train until every session PR "settled," and settled included non-merged states (`blocked:ci`, rebase-blocked, rate-limit-capped, still-pending at the ceiling) — so a session could report "done" with red/DIRTY/unmerged PRs still open. Phase b splits the one condition into two: **dispatch-pool-empty** (all queues empty + live backlog no net-new) is the *drain-entry gate* that ends the dispatch loop, while **session completion** is the stronger, drain-owned assertion — every open workable issue closed or dispositioned (merged, or labeled to a genuine human/operator class: `needs-human-review` / `needs-operator` / `needs-triage`) AND every session PR merged, with the *only* legitimate residual stopping point being a confirmed external dependency the agent cannot perform (a human-supplied secret, an upstream release, a provider-console action). The drain's job is reframed as **driving the tail** toward those completing states; every other "settled" class is now an explicit **bounded-exit concession** (stops the drain from spinning forever, but leaves the session incomplete and surfaces the PR for the next run) rather than a completion. The concrete auto-heal / dependency-update / release-arming *step mechanics* remain phase c (#663); this phase defines the contract they satisfy. Spec/docs-only; no runtime/helper change. Per issue-work scope discipline the *enforcement* of the three escalation actions (`file-tracking-issue`, `stop-auto-rerunning`, `apply-blocked-ci`) is deliberately deferred to a follow-up phase ([#385](https://github.com/mattsears18/shipyard/issues/385)) — `crossed` emits the configured action set on each crossed row so the future enforcement phase can act on it, but this phase files no issues, writes no flake-suspects file, and labels no PRs. Minor bump (new feature, additive — defaults preserve pre-#378 behavior).

### 1.5.29 — 2026-05-29

Closes [#374](https://github.com/mattsears18/shipyard/issues/374) (P2, bug) — the **end-of-session drain's 120-min wall-clock ceiling exited prematurely on healthy-but-slow merge trains**. The old ceiling fired on the wrong signal: a multi-PR session with full E2E sharding (3 shards × ~10-15 min), Lighthouse (~9 min), and release-please cascades (each merge re-DIRTY-ing siblings, then a 15-25 min rebase + re-CI round-trip) can legitimately take >2h to land even when nothing is wrong, and the cap cut those sessions off mid-flight — forcing the user to re-invoke `/do-work` to resume the drain (which also resets cost tracking). The fix replaces wall-clock with a **per-PR progress signal**: drain tracks each open PR's `headRefOid` and the timestamp it last moved (`head_unchanged_since`), and a PR counts as "settled" once its head has been unchanged for `ci.settled_minutes` (default 20) — CI grinding but the branch not being actively reworked. Drain exits the moment every session_pr is settled (merged/closed, `blocked:ci`, rebase-blocked, rate-limit-capped, or pending-head-quiet), with no fixed-poll no-progress window. A new `ci.max_drain_hours` (default 8h) survives as the belt-and-braces outer bound so a degenerate session (runaway CI loop, infinite rebase whose head moves every poll) still can't run forever. The per-poll status line now surfaces the progress clock per still-pending PR (`PR #N: head unchanged 12m / 20m settled-threshold`) so the operator can watch the signal that determines settle. Docs/spec + config-schema + test change — the drain loop is orchestrator-interpreted prose, so the behavior change lands in `drain.md` rather than a helper script. Companion to #373 (cross-session DIRTY adoption into `session_prs`, shipped earlier this session).
