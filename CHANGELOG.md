## shipyard

### 1.8.1 — 2026-05-30

Closes [#394](https://github.com/mattsears18/shipyard/issues/394) (P2, gap 1) — **the `do-work-split` thin-entry line cap now auto-derives from the orchestrator-state struct count instead of being a hardcoded magic number**. The `thin entry stays <= N lines` assertion in `do-work-split.test.sh` had to be manually re-baselined on every PR that adds an orchestrator-state struct to `commands/do-work.md` (200 → 220 → 222 → 223 across #195/#233/#246/#323/#387). Each bump was a separate edit a struct-addition PR could forget — and when it did, CI went red; if that PR merged anyway, `main` went red and a `fix-main-ci` divert burned a recovery cycle (the exact 1.7.2 sequence). The fix counts the documented top-level struct bullets (`- **\`name\`**`) in the `## Orchestrator state` section and asserts a derived cap of `197 base + 2 per struct` (currently 14 structs → 225, actual 223), so adding a 15th struct raises the cap by 2 automatically with no manual edit. The #154 intent is preserved: the base does NOT grow with struct count, so the cap stays fixed against non-struct prose — a thin entry that balloons with non-struct content still fails the assertion ("the split is over-engineered, we'd rather know"). Patch bump (test-tooling change — no plugin runtime behavior changed). Gap 2 from the issue (making `Shell scripts (lint + tests)` a required status check on the default-branch ruleset) is a GitHub branch-protection change, not a code change, and remains a human follow-up.

- **`plugins/shipyard/scripts/tests/do-work-split.test.sh`** — section (2) replaces the hardcoded `assert_line_count_at_most "$do_work_path" 223` with a struct-derived cap computed from `STRUCT_BASE (197) + STRUCT_PER_BUDGET (2) × <struct_count>`, where `struct_count` is the count of top-level `- **\`name\`**` bullets scoped to the `## Orchestrator state` section; new section (2b) pins a `struct_count >= 10` floor so a heading rename or struct-format change can't silently collapse the derivation back to a de-facto magic number; top-of-file comment + the new derivation comment explain the mechanism and cite #394.
- **`plugins/shipyard/.claude-plugin/plugin.json`** — version bump 1.8.0 → 1.8.1 (patch — test tooling).

### 1.8.0 — 2026-05-30

Closes [#391](https://github.com/mattsears18/shipyard/issues/391) (P2, feat) — **`/shipyard:my-turn` now leads with the single next action instead of a full ranked list**. The command's tagline promises *focus* ("what's the ONE thing blocking the most, that I should do next") but its v1 output rendered a 20-line ranked backlog by default, reintroducing exactly the prioritization burden the command was meant to remove — dogfooding on `mattsears18/lightwork` (2026-05-30) surfaced this as reading like an "issue dump" rather than "your next step." This adopts Proposal A from the issue: the default render is now just the #1-ranked item as a prominent `→ Next:` directive (with its URL and any dependency/unblocks context, since "what this unblocks" is part of "what to do next"); the full ranked list is opt-in via a new `--all` flag (or an explicit `--limit N > 1`). The 0-items empty-state one-liner is unchanged — it's identical across modes. Minor bump (new default behavior + new `--all` flag, additive).

- **`plugins/shipyard/commands/my-turn.md`** — frontmatter description + `argument-hint` updated to advertise the single-next-action framing and the new `--all` flag; new **Mode resolution** subsection under Args defining single-action mode (default) vs. list mode (`--all` or `--limit N > 1`, with `--limit 1` equivalent to the default); Output section restructured to lead with the **Single-action mode** render (the `→ Next:` directive, its URL, an optional dependency/unblocks line, and a `<K> more — rerun with --all` remainder footer) followed by the existing list render as **List mode**; empty-state marked unchanged-across-modes; limit-overflow scoped to list mode; new **Don't** entry forbidding the default full-list dump.
- **`plugins/shipyard/scripts/tests/my-turn.test.sh`** — five new regression assertions guarding the single-action default (`→ Next:` directive present, `--all` flag accepted, single-action + list modes documented, empty-state one-liner preserved).
- **`plugins/shipyard/.claude-plugin/plugin.json`** — version bump 1.7.2 → 1.8.0 (minor — new feature).

### 1.7.1 — 2026-05-29

Closes [#387](https://github.com/mattsears18/shipyard/issues/387) (P1, bug) — the **harness `isolation: "worktree"` dispatch path leaks `do-work/*` branch checkouts into the user's PRIMARY working tree**, stranding the drain. The orchestrator runs exclusively in its own `.claude/worktrees/orchestrator-<id>` worktree and never issues `git checkout do-work/*` against the primary, yet the primary HEAD reflog shows leaked checkouts (`moving from main to do-work/issue-378`, etc.) originating from the harness and/or a dispatched agent operating against the shared `.git`. The damage surfaces at drain: a leaked `do-work/*` checkout holds git's per-branch lock on a PR head branch, so a drain-phase `fix-rebase` worker can't `git switch <head>` ("already checked out at <primary>") and bails `blocked rebase` — defeating drain's whole purpose. Root cause is harness behavior shipyard can't change, so the fix is an orchestrator-side defensive guard. Patch bump (spec/docs change — the orchestrator loop is interpreted prose, no helper script touched).

- **`plugins/shipyard/commands/do-work/steady-state.md`** — new step **A.0.6** (primary-checkout branch-leak guard), fired every reconcile turn before A.1: resolves the primary checkout from the orchestrator-worktree path, and if the primary is off the default branch, restores it (`git -C <primary> checkout <default> && pull --ff-only`) when the tree is clean (lossless) or surfaces a loud `[primary-leak] WARNING` and skips when the tree is dirty (possible real edits — never auto-restore). Increments `primary_leak_restores` / `primary_leak_dirty_skips`. This is the single source of truth for the guard.
- **`plugins/shipyard/commands/do-work/drain.md`** — runs the A.0.6 guard once at **drain entry** (where the leak does its damage), and extends the [pre-dispatch head-branch reap](https://github.com/mattsears18/shipyard/issues/370) to detect the **primary** holding the head branch (restore-if-clean / warn-if-dirty — never reap the primary), as a belt-and-suspenders for a leak that lands mid-drain.
- **`plugins/shipyard/commands/do-work.md`** — new `primary_leak_counters` orchestrator-state struct (`{ primary_leak_restores, primary_leak_dirty_skips }`), session-local, initialized all-zero, on the `ci_session_counters` precedent.
- **`plugins/shipyard/commands/do-work/dont.md`** — documents the one sanctioned exception to "never write to the primary checkout": the corrective `git -C <primary> checkout <default>` the guard runs only on the leak-detected-and-clean path.
- **`plugins/shipyard/commands/do-work/cleanup-summary.md`** — new `Primary-checkout leaks (#387)` end-of-session friction line (omitted when both counters are zero); a non-zero dirty-skip is the actionable signal (primary still parked on a `do-work/*` branch with uncommitted work).
- **`plugins/shipyard/.claude-plugin/plugin.json`** — version bump 1.7.0 → 1.7.1 (patch — bug fix).

### 1.6.0 — 2026-05-29

Closes [#378](https://github.com/mattsears18/shipyard/issues/378) (P1, feat) — **phase 1 of the cross-PR flake registry**. Today's `fix-checks-only` worker handles flakes per-PR in isolation: none of the independent workers sees that the SAME test has flaked on N other PRs this week, so chronic flakes go uninvestigated and the user's repo-level rule ("Flaky CI is a real failure — fix the root cause") is silently violated. This phase ships the **data layer + threshold tooling**: a session-spanning `~/.shipyard/flake-registry.jsonl` ledger, a `fix-checks-only` write path that records one event per flake conclusion (gated on `flake_registry.enabled`, default off), the `shipyard.config.json::flake_registry` config schema (window / rerun-threshold / distinct-PRs-threshold / actions), and a `flake-registry.sh crossed` reader that aggregates per-(workflow,job,test) flake rates over the window and names which flakes have crossed the escalation bar. Per issue-work scope discipline the *enforcement* of the three escalation actions (`file-tracking-issue`, `stop-auto-rerunning`, `apply-blocked-ci`) is deliberately deferred to a follow-up phase ([#385](https://github.com/mattsears18/shipyard/issues/385)) — `crossed` emits the configured action set on each crossed row so the future enforcement phase can act on it, but this phase files no issues, writes no flake-suspects file, and labels no PRs. Minor bump (new feature, additive — defaults preserve pre-#378 behavior).

- **`plugins/shipyard/scripts/flake-registry.sh`** (new) — the ledger helper, modeled on `cost-history.sh`. Subcommands: `record` (append one event: repo/pr/workflow/job + optional test/action/session, append-only JSONL safe under concurrent writers), `report` (windowed per-(workflow,job,test) aggregation: event count, distinct-PR count, PR list, first/last timestamps), `crossed` (threshold filter — event count ≥ `rerun_threshold` AND distinct-PR count ≥ `distinct_prs_threshold` within `window_days`; echoes the configured action set per row), `read` / `prune` / `reset` housekeeping. GNU+BSD `date` fallback for the cutoff math; corrupt lines dropped defensively at read.
- **`plugins/shipyard/scripts/tests/flake-registry.test.sh`** (new) — 24 assertions covering record arg-validation + dry-run + optional-field omission, windowed aggregation (in/out-of-window, repo isolation), the threshold + distinct-PRs guard (5 events on 1 PR does NOT cross), custom `--actions` echo, prune, and reset-to-.bak.
- **`plugins/shipyard/schemas/shipyard.config.schema.json`** + **`plugins/shipyard/scripts/shipyard-config.sh`** — new `flake_registry` config block (`enabled` default `false`, `window_days` 7, `rerun_threshold` 3, `distinct_prs_threshold` 2, `actions` enum-constrained to the three escalation actions). Added to both the schema (with `additionalProperties: false` + per-field descriptions) and the built-in defaults so every consumer sees the value on a bare repo.
- **`plugins/shipyard/agents/issue-worker/fix-checks-only.md`** — new "Record a flake event in the cross-PR flake registry" section after the root-cause-context step: gates on `flake_registry.enabled`, documents when to record (genuine flakes only — not real-bug fixes), the `flake-registry.sh record` call shape, and the explicit phase-1 scope boundary (record only; no enforcement yet).
- **`plugins/shipyard/.claude-plugin/plugin.json`** — version bump 1.5.28 → 1.6.0 (minor — new additive feature).

### 1.5.29 — 2026-05-29

Closes [#374](https://github.com/mattsears18/shipyard/issues/374) (P2, bug) — the **end-of-session drain's 120-min wall-clock ceiling exited prematurely on healthy-but-slow merge trains**. The old ceiling fired on the wrong signal: a multi-PR session with full E2E sharding (3 shards × ~10-15 min), Lighthouse (~9 min), and release-please cascades (each merge re-DIRTY-ing siblings, then a 15-25 min rebase + re-CI round-trip) can legitimately take >2h to land even when nothing is wrong, and the cap cut those sessions off mid-flight — forcing the user to re-invoke `/do-work` to resume the drain (which also resets cost tracking). The fix replaces wall-clock with a **per-PR progress signal**: drain tracks each open PR's `headRefOid` and the timestamp it last moved (`head_unchanged_since`), and a PR counts as "settled" once its head has been unchanged for `ci.settled_minutes` (default 20) — CI grinding but the branch not being actively reworked. Drain exits the moment every session_pr is settled (merged/closed, `blocked:ci`, rebase-blocked, rate-limit-capped, or pending-head-quiet), with no fixed-poll no-progress window. A new `ci.max_drain_hours` (default 8h) survives as the belt-and-braces outer bound so a degenerate session (runaway CI loop, infinite rebase whose head moves every poll) still can't run forever. The per-poll status line now surfaces the progress clock per still-pending PR (`PR #N: head unchanged 12m / 20m settled-threshold`) so the operator can watch the signal that determines settle. Docs/spec + config-schema + test change — the drain loop is orchestrator-interpreted prose, so the behavior change lands in `drain.md` rather than a helper script. Companion to #373 (cross-session DIRTY adoption into `session_prs`, shipped earlier this session).

- **`plugins/shipyard/commands/do-work/drain.md`** — replaced the "Hard ceiling: 120 min" rule with the progress-based termination criterion + `max_drain_hours` ceiling; added the `head_unchanged_since` per-session structure and the `settled_minutes` / `max_drain_hours` config reads at drain entry; redefined `P_settled` and the settled definition around the per-PR head-movement threshold; added the per-PR progress-detail status line. Updated the intro, soft-drain step 5, and per-poll bookkeeping step 3 to match.
- **`plugins/shipyard/schemas/shipyard.config.schema.json`** — new `ci.settled_minutes` (integer, min 1, default 20) and `ci.max_drain_hours` (number, min 0.1, default 8) keys; `ci` block description notes the #374 keys ship with active non-zero defaults (they replace a hardcoded ceiling) vs the #323 opt-in gates.
- **`plugins/shipyard/scripts/shipyard-config.sh`** — built-in defaults for the two new `ci` keys.
- **`plugins/shipyard/commands/do-work.md`**, **`commands/do-work/setup.md`**, **`commands/do-work/cleanup-summary.md`**, **`commands/do-work-RATIONALE.md`** — updated the cross-references to the drain ceiling (struct-list settle note, C=1 drain-semantics line, `Drain phase` summary reason strings, and the RATIONALE's why-progress-based-not-wall-clock paragraph).
- **`plugins/shipyard/scripts/tests/shipyard-config.test.sh`** — default-value + round-trip + below-minimum-rejection assertions for the two new keys (115 assertions).
- **`plugins/shipyard/scripts/tests/do-work-split.test.sh`** — asserts drain.md carries `head_unchanged_since` / `settled_minutes` / `max_drain_hours` and no longer carries the "Hard ceiling: 120 min" behavioral rule (129 tests).
- **`plugins/shipyard/.claude-plugin/plugin.json`** — version bump 1.5.28 → 1.5.29 (patch — spec/docs + config + test change closing a drain-duration friction gap).
