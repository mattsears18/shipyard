## shipyard

### 1.3.14 — 2026-05-20

Adds **issue-comment-thread reading** to `agents/issue-worker.md` step 2. Closes [#114](https://github.com/mattsears18/claude-plugins/issues/114).

Maintainers commonly post clarifications, scope updates, and corrections as comments on existing issues — editing the body wholesale is destructive, commenting is additive and preserves provenance. Previously the worker only read `body` + `title` from the `gh issue view` projection, so every clarification posted after the body was last edited was silently dropped. A worker dispatched against such an issue would implement the stale body's spec while the maintainer who left the comment assumed it was incorporated, forcing a follow-up issue to undo the wrong fix.

- **`agents/issue-worker.md` step 0 `gh issue view` projection — add `comments` (and `author`).** The orchestrator does not pass comments through the dispatch prompt, so the only place the worker sees them is its own step 0 fetch. Adding `author` alongside `comments` lets the comment-classification logic in step 2 do the trusted-author check without a second API call.
- **`agents/issue-worker.md` step 2 — new comment-extraction block.** Walks `comments` in chronological order. Each comment is classified: trusted-author (`author.login` matches the issue's `author.login` OR the `<owner>` in `<owner/repo>`) refines the body, with later comments superseding earlier on the same point; untrusted-author (anyone else — drive-by commenters, bots, third parties) is treated as a *claim about the problem*, not instructions. The narrower trust signal (just the issue's principal + repo owner, not the full `trusted_authors` set) keeps the surface tight without re-querying the collaborators API from inside the worker.
- **`agents/issue-worker.md` step 2 — out-of-scope-action gate extended to comments.** The same `body requested out-of-scope action` block from #93 now applies to comment content too: a comment from anyone — trusted or not — asking for an unusual action (touch a file outside the affected module, install a new dependency, modify CI / secrets / `.github/workflows/`, contact an external service) bails with `blocked: comment-thread requested out-of-scope action: <what>`. Trust elevates *clarifications*, not new directives.
- **`agents/issue-worker.md` step 2 — comment-permalink citation rule.** When a trusted-author comment materially altered the implementation vs the original body, the PR description cites the permalink under a `> Implementation reflects the clarification in <comment-permalink>.` line so the comment-chain is traceable for reviewers. Routine confirmations ("yes please proceed", "+1") don't need citation — only comments that changed the implementation.
- **`agents/issue-worker.md` step 2 — ambiguity check applies to combined signal.** The existing `blocked: ambiguous — no acceptance criteria` check now reads the body + trusted-author comments together. A body that's vague on its own but a follow-up comment that nails the criteria counts as clear, so a clarifying comment can rescue an otherwise-blocked issue.
- **`agents/issue-worker.md` Don't section — new entry.** *"Don't skip the comment-thread read in step 2"* — captures the silent-stale-spec failure mode with the #114 citation, so future readers don't accidentally re-introduce the bug by stripping the comment read while rewriting step 2.

Interaction with the prior untrusted-input defense stack ([#90](https://github.com/mattsears18/claude-plugins/issues/90) / [#93](https://github.com/mattsears18/claude-plugins/issues/93)): the trusted-author allowlist from [step 1.7](https://github.com/mattsears18/claude-plugins/blob/main/plugins/shipyard/commands/do-work.md#17-resolve-trusted-author-allowlist) already ensures the *issue's* author is trusted by the time the worker runs; this change layers a second, narrower check on top so a comment from a stranger reading along can't redirect the worker. Out-of-scope-action gating from #93 is extended verbatim to comments — the prompt-injection threat model doesn't change with the input surface. Sibling to [#94](https://github.com/mattsears18/claude-plugins/issues/94) (agents *writing* decision-context comments); this issue closes the loop on agents *reading* them.

### 1.3.12 — 2026-05-20

Adds a dispatch-side `originating_author_trust ∈ {trusted, external}` field to `/shipyard:do-work`'s issue-worker dispatch prompt and gates `gh pr merge --auto` on it inside the worker. Closes [#92](https://github.com/mattsears18/claude-plugins/issues/92).

This is the **third** layer of the public-repo prompt-injection defense, after the intake-side `needs-human-review` auto-label workflow ([#91](https://github.com/mattsears18/claude-plugins/issues/91)) and the dispatch-side trusted-author allowlist (1.3.9 / [#90](https://github.com/mattsears18/claude-plugins/issues/90)). In normal operation a stranger-authored issue is already filtered out by both — it carries `needs-human-review` from intake (so step 4's label filter excludes it) and lands in bucket 0.5 at dispatch (so the client-side filter drops it). The auto-merge gate fires only when both principal gates have failed simultaneously: an external issue somehow gets all the way to a worker, the worker still labels the PR `needs-human-review` and skips `gh pr merge --auto`, so the PR sits waiting for a maintainer's manual merge instead of riding auto-merge straight to `main`.

- **`commands/do-work.md` step 7 dispatch rules — new author-trust computation block.** Before composing each dispatch prompt: `originating_author_trust = "trusted"` if the candidate's `author.login` (lowercased) is in `trusted_authors` (the same set populated by [step 1.7](https://github.com/mattsears18/claude-plugins/blob/main/plugins/shipyard/commands/do-work.md#17-resolve-trusted-author-allowlist) in 1.3.9), else `"external"`. The dispatch prompt template names the field explicitly with the form *"the originating issue's author trust is **`<trust>`**"*. Reuses 1.3.9's `trusted_authors` cache rather than re-fetching collaborators — single source of truth.
- **`commands/do-work.md` step 4 explanation updated.** The `author` field on the `gh issue list` payload now serves both 1.3.9's client-side filter AND 1.3.12's downstream dispatch-time computation — the explanation under the fetch command enumerates both uses so the field's provenance is documented end-to-end.
- **`agents/issue-worker.md` Inputs section — new field documentation.** Names `originating_author_trust` as a normal-mode-only field, names the load-bearing step (step 6 auto-merge), and codifies the fail-safe — if the field is missing or unparseable from the dispatch prompt, the worker treats it as `external`. Fix-checks / fix-rebase / fix-main-ci / fix-failing-prs-batch dispatches don't carry the field (they don't open new originating-issue-scoped PRs).
- **`agents/issue-worker.md` step 6 — conditional auto-merge.** When `trusted`: `gh pr merge --auto --merge --delete-branch` as before. When `external`: skip `gh pr merge` entirely, run `gh pr edit --add-label needs-human-review`, post a comment citing #90 / #91 / #92 / [#93](https://github.com/mattsears18/claude-plugins/issues/93) so the gate's provenance is visible to the maintainer. The external branch is the gate this release exists to enforce. Sits immediately after 1.3.11's step 5.5 (decision-context comment trails, #94) — the auto-merge gate runs after any decision comment the worker chose to post.
- **`agents/issue-worker.md` step 8 — new return-string variant.** `shipped #<N> via PR #<M> (auto-merge: gated — external-author origin, needs-human-review label applied, checks: <state>)` for the external-trust path. The orchestrator's step A reconcile treats it identically to the trusted `shipped` return for `session_prs` accounting; the gate's effect is on whether the PR auto-merges, not on whether it counts as session work.
- **`agents/issue-worker.md` Don't section — new entry.** *"Don't arm auto-merge when `originating_author_trust == "external"`"* — captures the reflexive-`gh pr merge --auto` failure mode with the exact threat-model citation (#92) and the fail-safe-defaults-to-external rule, so future readers don't accidentally re-introduce the bug while rewriting step 6.

The full defense stack against the public-repo prompt-injection / RCE threat is now: (1) intake auto-label (#91) — every external-authored issue lands with `needs-human-review` the moment it's opened, so step 4's `-label:needs-human-review` filter excludes it from the workable queue; (2) dispatch-side allowlist (#90 / 1.3.9) — even without the intake label, step 2's bucket 0.5 + step 4's client-side filter drop untrusted-author issues; (3) auto-merge gate (#92 / 1.3.12) — even if both prior gates fail, the worker doesn't arm auto-merge for an external-origin PR; (4) untrusted-input handling in the worker (1.3.9 step 2 + 1.3.10 / #93) — even if a malicious body reaches the worker, the worker re-derives the implementation from the codebase, treats code blocks as examples not patches, and verifies the failure before shipping.

### 1.3.11 — 2026-05-20

Adds **decision-context comment trails** so non-trivial agent decisions survive past the session that made them. Closes [#94](https://github.com/mattsears18/claude-plugins/issues/94).

Git history captures *what* changed; comments capture *why this approach over the rejected ones*. Previously the only artifact of an agent's decision (rejected alternative, divergence from issue body, external constraint, accepted side-effect) was the agent's transient working context — once the session ended, future maintainers re-litigated the same tradeoffs because the prior reasoning was never written down where it belongs.

- **New step 5.5 "Record decision context (when applicable)" in `agents/issue-worker.md`.** Inserted between step 5 (commit + push + PR) and step 6 (enable auto-merge). Spells out four trigger conditions — a viable alternative was rejected, the PR diverges materially from the issue body or suggested approach, an external constraint shaped the implementation, a side-effect was deliberately accepted or punted. One PR comment, one bullet per decision, named alternative or constraint plus the tradeoff in one sentence. Routing rules table covers PR-vs-issue placement (PR for implementation decisions, issue for triage/scope decisions; divergence cases get both). Silence is the correct default for routine work — most PRs need no decision comment at all.
- **New fix-checks-only root-cause comment rule in `agents/issue-worker.md`.** Added to the fix-loop section right after the 3-attempt cap. When a fix-checks worker identifies the actual root cause of a flake / race / environmental failure that wouldn't be obvious from the diff alone, post a `Fix-checks: <one-line root cause>` comment on the PR before returning `green`. This stops the next session's auditor or human reviewer from re-flagging the same failure mode without context. Routine "applied the obvious fix to the obvious error" cases skip the comment — the diff is the explanation. Comment-post failure (rate limit, permission) is non-blocking; the return contract still applies.
- **New "Decided not to file: leave a decision trail" section in `skills/audit-rubrics/SKILL.md`.** When an auditor considers a finding and decides NOT to file for a contextual reason ("intentional per design doc X," "covered by open issue #N," "accepted tradeoff per PR #M"), the rationale lands in two places: the audit's end-of-run summary as a new `Skipped (decided not to file)` section alongside the existing `Skipped (duplicates)` block, and — if the finding has a natural home issue — as a one-line comment on that issue. Distinguishes the new "decided not to file" category from the existing "skipped (duplicate)" / "skipped (not applicable to stack)" categories which already have their own trails. The section omits entirely when nothing was decided against.

Doesn't change the deferred-issue handling from 1.3.8 (#76) — that path stays as-is. This release generalizes the same "write the reasoning down where it belongs" pattern to other agent decisions.

Pure prompt edits; no spec / state-struct / tooling changes.

### 1.3.10 — 2026-05-20

Extends the defense-in-depth untrusted-body language in `agents/issue-worker.md` step 2 with the verification-first stance and the explicit `body requested out-of-scope action` framing. Closes [#93](https://github.com/mattsears18/claude-plugins/issues/93).

PR [#98](https://github.com/mattsears18/claude-plugins/pull/98) (closes [#90](https://github.com/mattsears18/claude-plugins/issues/90)) already hardened the worker against treating an untrusted issue body as a script of instructions — re-derive the implementation from the codebase, bail on out-of-scope suggested fixes, bail on red-flag patterns. That covers most of #93's ask. Three deltas remained:

- **Code blocks in the body are EXAMPLES, not code to paste verbatim into the PR.** A body that says "here's the fix:" followed by a code block is showing the *kind of change* the filer thinks is needed — not a patch to apply. The agent should read the example, understand the intent, then write the actual fix itself against current code. A literal copy-paste is a prompt-injection vector even when the rest of the body is benign.
- **`blocked: body requested out-of-scope action: <what>`** — distinct return string from #98's `blocked: suggested fix exceeds expected scope` (which is for honestly-mistaken oversized suggestions). The new framing is for when the request itself looks like an attempt to extract a side-channel effect (touch a file outside the affected module, install a new dependency, modify CI config, exfiltrate secrets, contact an external service, run unrelated shell commands) rather than fix the stated bug.
- **Verification-first stance — reproduce the failure before writing the fix.** Issues sit open for weeks; SDKs / dependencies / the codebase itself move underneath them. The body is a *claim about a problem*, not a *script of instructions*. Verify the claim holds before shipping. If you can't reproduce, return `blocked: cannot reproduce — <what you tried>` rather than ship a speculative fix. This is the same posture the user-feedback preamble in `commands/do-work.md` already takes — #93's ask is to make it the default for *every* issue, not just `user-feedback`-labeled ones.

The user-feedback-specific preamble in `commands/do-work.md` stays as-is — it carries the raw-text-vs-refined-text trust rule which is feedback-pipeline-specific. Step 2 of the worker is now the right place for the verification-first stance, which applies universally.

Single-file prompt edit; no spec / state-struct / tooling changes.

### 1.3.9 — 2026-05-20

Adds a **trusted-author allowlist** to `/shipyard:do-work` so strangers' issues on public repos can no longer be dispatched against by issue-workers. Closes the P0 prompt-injection / RCE-class hole documented in [#90](https://github.com/mattsears18/claude-plugins/issues/90): any GitHub user could open an issue on a public repo without any of the `/do-work` skip-labels and the next session's issue-worker would read the body as instructions, ship a PR, and arm auto-merge — landing attacker-controlled code in `main` on the maintainer's machine if CI passed.

- **New step 1.7 "Resolve trusted-author allowlist" in `commands/do-work.md`.** Runs after step 1.5 (dashboard launch) and before step 2 (backlog overview). Resolution order: `.shipyard/trusted-authors.txt` per-repo override file first (one login per line, comments OK, case-insensitive; repo owner always implicitly trusted), then fallback to the live `gh api repos/<owner/repo>/collaborators` query filtered to push-permission collaborators, then conservative fallback to just the repo owner if the API call errors. Bot accounts (`dependabot[bot]`, `github-actions[bot]`, etc.) are NOT auto-trusted — the maintainer adds them to the override file explicitly if they want bot-filed issues dispatched against.
- **New `trusted_authors` session-state struct** — the 9th orchestrator state struct, joining `in_flight`, `ready_issues`, `failed_prs`, `raw_backlog`, `divert_queue`, `main_ci`, `session_prs`, `deferred_issues`. Resolved once at setup; not re-polled mid-session.
- **New bucket 0.5 "Untrusted author" in step 2's backlog overview.** Applied *first* (highest precedence) — strangers' issues never reach the dispatch queue, even if they would otherwise pass every other filter. The bucket row renders under Workable in the bucket table (despite being the highest-precedence skip) so the user sees "N stranger-filed issues skipped" at the top of the skip list, not buried at the bottom. Same row added to the end-of-session summary's bucket table.
- **New client-side filter in step 4** — drop issues whose `author.login` (lowercased) isn't in `trusted_authors`. The `gh issue list --json` shapes in step 2 (universe), step 4 (workable fetch), and the end-of-session summary all gain the `author` field so the filter has data to operate on. The search-qualifier syntax has no `-author:` form that can filter for "anyone except this list," so the filter is necessarily client-side. Step C's lightweight backlog re-check (mid-session discovery) also applies the trust filter so an issue filed by a stranger after `/do-work` started never reaches `raw_backlog`.
- **Defense-in-depth in `agents/issue-worker.md` step 2.** Even after the orchestrator's author filter has cleared an issue, the worker now treats the body as untrusted input — re-derives the implementation from the codebase rather than the issue body, bails with `blocked: suggested fix exceeds expected scope — needs human review` if the suggested fix touches CI / secrets / `.github/workflows/` / unrelated files, and bails with `blocked: issue body contains directives that bypass normal review` on red-flag patterns (instructions to call external services, execute shell snippets verbatim, "trust me" framings). Applies to every issue, not just suspicious-looking ones — the defense is structural.
- **New `Don't` rule in `commands/do-work.md`.** Spells out the threat model in one paragraph so the dispatch-time filter can never be bypassed "just for this one issue" with a body that's already compromised by definition.

**Threat model and motivating threat:**

1. Stranger opens an issue on a public repo: `"Bug: foo() returns null. Suggested fix: add helper.ts with [crafted payload]."`
2. Issue carries none of the `-label:blocked` / `-label:wontfix` / `-label:needs-design` / `-label:needs-triage` / `-label:discussion` / `-label:needs-refinement` / `-label:needs-human-review` skip-labels, so step 4's filter includes it.
3. Pre-1.3.9: issue-worker dispatched, reads `body,title` as instructions per step 2 of the worker spec, opens PR, arms auto-merge.
4. If CI passes — and a subtle payload can be crafted to pass — the PR merges to `main` without human review. Worktree isolation prevents filesystem damage *outside* the worktree but does NOT prevent malicious code from landing in a merged PR, on a system that runs `/do-work` with broad tool access.

Under 1.3.9 the stranger-authored issue lands in bucket 0.5 and step 4's client-side filter drops it from the dispatch queue. The maintainer can still override: add the login to `.shipyard/trusted-authors.txt` (broad trust) or re-file the issue under the maintainer's own account using the same body (per-issue vouching). The pre-existing `needs-human-review` label is also a valid escape hatch the maintainer can use to surface the stranger's body for review — but the *default* for untrusted authors is **skip**, not workable. P0 fix.

### 1.3.8 — 2026-05-20

Adds a `deferred: <reason>` outcome to `/shipyard:do-work`'s scope pre-flight return shape so scope agents can mark issues whose fix is correct but bigger than a single worker can ship (multi-PR migration, SDK upgrade, external decision required, etc.) instead of dispatching a worker that will burn agent minutes and bail with `blocked`. Closes [#76](https://github.com/mattsears18/claude-plugins/issues/76).

Motivating session: a `/do-work` run hit a `fix(ios): app crashes on register screen — EXC_BAD_ACCESS` issue whose scope-preflight diagnosis was an Expo SDK bump (multi-PR effort). The orchestrator dispatched a worker that bailed with `blocked` after ~10 min — wasted agent time on un-shippable work. Under 1.3.8 the scope agent returns deferred and the slot goes to the next workable item.

### 1.3.7 — 2026-05-20

Makes the live dashboard launch + per-turn rewrite enforceable in `/shipyard:do-work` so the orchestrator can no longer silently skip them across an entire session. Closes [#78](https://github.com/mattsears18/claude-plugins/issues/78).

- **New numbered step 1.5 in `plugins/shipyard/commands/do-work.md`.** The launch recipe — initial HTML write, `open /tmp/do-work-dashboard.html`, `nohup`-detached updater spawn — was hoisted out of the prose "Launch sequence" subsection of the "Live dashboard" section and into its own numbered Setup step ("1.5 Launch the live dashboard") between step 1 and step 2. ### 1.3.6 — 2026-05-20

Augments `/shipyard:do-work`'s step 2 backlog overview with **inline action-recommendation sub-rows** alongside the existing raw bucket counts, so stale-skipped issues surface with concrete next steps the human can act on instead of being buried in counts. Closes [#75](https://github.com/mattsears18/claude-plugins/issues/75).

Previously a session whose `Workable` count was 0 — but whose Skipped buckets contained issues whose blockers had since closed, or `needs-triage` issues that were actually already triaged (P-label + type label + acceptance criteria + repro) — would terminate as "backlog empty" with no signal to the user that the backlog had triageable residue. The user had to manually audit the skipped buckets before each session. The 1.3.1 release shipped step 3d.2's `blocked`-label auto-clear sweep, but that fires after step 2 has already printed and only addresses the mechanical "all referenced blockers closed" case. This release adds the same kind of visibility for `needs-triage` issues (which step 3d.2 doesn't touch) and gives the user a pre-sweep view of which `blocked` issues are about to auto-clear.

- **New "Inline action-recommendation candidates per skipped bucket" section in `plugins/shipyard/commands/do-work.md` step 2.** Documents the two surfaces:
  - **Bucket 6 (`Blocked` label) → `likely-clearable` candidates.** Parse `Blocked by #N` references from each blocked issue's body (same regex step 3d.2 uses) and check each referenced blocker's state. Surface issues where ALL referenced blockers are `CLOSED`/`MERGED` as candidates. These overlap with what step 3d.2 will auto-clear later — the step 2 surface gives the user pre-sweep visibility to intervene if any look wrong.
  - **Bucket 5 (`Needs triage / design`) → `likely-triageable` candidates.** Score each issue by five mechanical triage signals (priority label present, type label present, `## Acceptance` section, `## Repro` section, no co-gate like `needs-design`/`needs-human-review`). Score `>= 3` → candidate for "review then remove `needs-triage`." The orchestrator does NOT auto-remove the label — the threshold is a heuristic and the user judges.
- **Updated example output shape.** The bucket table (introduced by 1.3.5 / #77) now carries the candidate counts as indented sub-rows under their parent bucket: `⚠ likely-clearable` / `⚠ likely-triageable` rows with the same `# / Issues` columns. Sub-rows omit entirely when their count is 0.
- **New edge case entries.** Action-recommendation sub-rows with zero candidates are omitted; the `likely-clearable` overlap with step 3d.2 is intentional and not redundant (two sequential checkpoints — step 2 for visibility, step 3d.2 for action). Cost is documented as O(blocked-issues × referenced-blockers) `gh issue view --json state` lookups (cacheable with step 3d.2's sweep into a shared `blocker_state` map) plus a pure regex scan over already-fetched bodies — well under 1s wall-clock on backlogs under ~50 issues.

### 1.3.3 — 2026-05-20

Makes the LOCKFILE rule in `/shipyard:do-work`'s dispatch logic **section-aware** so multiple workers touching disjoint `package.json` sections (e.g., `overrides` vs `dependencies` vs `scripts`) can co-run. Closes [#59](https://github.com/mattsears18/claude-plugins/issues/59).

Previously the LOCKFILE rule was a blanket "if `touches_lockfile`, the rest of the pool parks." In LOCKFILE-heavy sessions — multiple dependabot bumps, security patches, scripts/coverage tweaks queued together — this dropped pool utilization to 1/N for the duration of every lockfile worker. In practice these PRs almost never actually conflict: `package.json` is a JSON object with disjoint top-level sections, and `package-lock.json` regenerates additively from the resolved tree post-merge. The all-park rule was overcautious.

- **`commands/do-work.md` dispatch rules — new section-aware lockfile rule.** `ready_issues` entries now carry `lockfile_sections: [...]` (e.g., `["overrides"]`, `["dependencies", "devDependencies"]`, `["scripts"]`, `["go.mod"]`) instead of the boolean `touches_lockfile` flag. The dispatch rule treats each section as an additional claim: a candidate is blocked by section collision only when at least one of its `lockfile_sections` appears in some in-flight worker's set. Disjoint sections co-run normally — three workers claiming `overrides`, `dependencies`, and `scripts` respectively all dispatch in parallel. Overlapping claims still park, so two workers both touching `overrides` are still serialized (correctly — they would clobber each other's edits to the same JSON object).
- **`commands/do-work.md` step 6 scope pre-flight — new return shape.** Scoping agents now return `{ issue: N, files: [...], lockfile_sections: [...] }`. For `package.json`-class manifests the sections are the standard top-level keys (`overrides`, `dependencies`, `devDependencies`, `peerDependencies`, `optionalDependencies`, `scripts`, `engines`, `config`, `workspaces`, `resolutions`, `pnpm`, etc.); for non-`package.json` lockfile-class files (`Gemfile`, `go.mod`, `Cargo.toml`, generated SQL migrations, root build config) the filename is the section token, coarser-grained but applying the same disjoint-claims-co-run principle. Empty array means the candidate isn't a lockfile-toucher and the section-aware check is a no-op for it.
- **`package-lock.json` / `pnpm-lock.yaml` / `go.sum` / `Cargo.lock` are NOT claimed as sections.** They regenerate additively post-merge — the merge train's existing fix-rebase worker already handles their textual conflicts by re-running the package manager against the rebased manifest (per the trivial-conflict-or-bail policy added in 1.2.9). Treating them as claims would re-introduce the all-park behavior this change is undoing.
- **`idle_reason` valid set gains `all ready_issues collide with in_flight lockfile sections (<section>×<N>, ...)`** to name the *sections* in flight rather than the toucher's issue number — since multiple lockfile-touchers may be in flight simultaneously on disjoint sections, naming a single `#N` toucher would be ambiguous. The pre-1.3.3 `lockfile-toucher #N in flight (other slots parked)` reason is removed from the valid set since it no longer matches the spec's behavior.
- **Live dashboard worker-card behavior updated.** When a lockfile-toucher is active, only show other slots as parked if their candidate's `lockfile_sections` overlaps the in-flight toucher's set. Sibling lockfile workers on disjoint sections render normally, not parked.
- **`Don't` rule rewritten.** "Don't dispatch alongside a lockfile-toucher" → "Don't dispatch a candidate whose `lockfile_sections` overlaps an in-flight worker's `lockfile_sections`." The rule still serializes overlapping claims (those would clobber the same JSON keys), but allows the common case of disjoint-section parallel work.

Motivating session: a `mattsears18/lightwork` `/do-work --concurrency 8` run had 3 LOCKFILE-class issues queued (#882: new firebase-admin dep in `dependencies`, #899: protobufjs DoS patch in `overrides`, #910: test-coverage tweak in `scripts` + `jest` config). Under the pre-1.3.3 rule, all three serialized at 1/8 utilization for the duration of each one. Under the section-aware rule, all three would have dispatched in parallel and the session would have completed in ~1/3 the time. The actual PRs merged cleanly without significant conflict at land time, confirming the section-disjoint hypothesis.

### 1.2.9 — 2026-05-20

Adds a drain-phase **fix-rebase** worker mode so the end-of-session drain can auto-rebase PRs whose only blocker is a stale base (`mergeStateStatus: DIRTY` with no failing checks). Closes [#61](https://github.com/mattsears18/claude-plugins/issues/61).

Previously the drain phase only dispatched fix-checks workers against newly-red PRs — DIRTY-but-green PRs sat untouched until the drain's no-progress termination criterion fired, leaving sessions to end with a tail of PRs needing manual rebase. After this change, the drain treats DIRTY PRs as actionable: a lightweight worker rebases onto current main, force-pushes with lease, and lets the merge train re-arm auto-merge naturally.

- **New `fix-rebase` mode in `plugins/shipyard/agents/issue-worker.md`.** Sits alongside `fix-checks-only` as a PR-targeted dispatch mode. The worker lands on the PR's head branch via the safe two-step (`git fetch origin <ref> && git switch <ref>`), pre-flights `mergeStateStatus` (bailing if no longer DIRTY or if hard failures appeared), runs `git rebase origin/<default-branch>`, applies a **trivial-or-bail conflict policy**, and `git push --force-with-lease`. The trivial-resolvable set is intentionally narrow: append-only conflicts in `CHANGELOG.md` / `CLAUDE.md` / `README.md` / shared docs, `ci.yml` shard-matrix appends, and lockfile entries that can be regenerated by re-running the package manager. Anything semantic (same function body / same imports / same JSON key with different values / file rename collisions / snapshot or fixture conflicts) hits `git rebase --abort` and returns `blocked rebase` immediately. The worker does NOT touch the PR title / body / labels, does NOT call `gh pr merge` manually (auto-merge was armed when the PR was opened and rebasing doesn't un-arm it), and does NOT `--watch` checks.
- **New drain-phase dispatch rule in `commands/do-work.md`.** Per-poll bookkeeping now tracks `D_dirty` (PRs in `session_prs` with `mergeStateStatus == "DIRTY"`, no hard-failure checks, not in flight, not yet in `rebase_blocked_prs`) alongside the existing `R_new` / `M_since_last` / `P_settled` sets. The drain dispatcher fills the pool with fix-checks workers first (red PRs are more urgent than stale-base PRs), then any remaining slots run fix-rebase workers — combined fix-checks + fix-rebase in-flight count stays under the session's `--concurrency` cap. Each fix-rebase is **one-shot per PR per session**: a `blocked rebase` return puts the PR into `rebase_blocked_prs` and the drain doesn't re-dispatch even if the PR stays DIRTY across subsequent polls. The drain's "settled" definition counts `rebase_blocked_prs` members as settled so a non-trivially-conflicted PR doesn't keep the drain alive indefinitely.
- **Drain status line gains visibility.** The per-poll `[drain] open=… newly_red=…` line now includes `dirty=<D_dirty>`, `rebase_blocked=<n>`, and a worker-count breakdown (`in_flight=<n>(fix-checks=<a>, fix-rebase=<b>)`) so the user can see at-a-glance how many PRs the rebase logic is working through.
- **End-of-session summary gains a `Drain-phase rebases` line** with the per-outcome breakdown (`<rebased_count> succeeded (#A, #B, ...)`, `<rebase_blocked_count> blocked (#C — <reason>, ...)`) — omitted entirely when both counts are zero so small sessions with no DIRTY churn stay terse. The `Drain phase` summary's `final session_prs state` partition also gains a `rebase-blocked: <r>` field alongside the existing `merged` / `ci-blocked` / `still pending` counters so the partition still sums to `|session_prs|`.

- **New `Don't` rules.** Two on the orchestrator side ("don't dispatch fix-rebase outside the drain," "don't retry a `blocked rebase` PR within the same session"), two on the worker side ("don't resolve a non-trivial merge conflict in fix-rebase mode," "don't `gh pr merge` manually in fix-rebase mode") — both pairs cite #61 as the motivating bug.

### 1.2.8 — 2026-05-20

Hardens the fix-checks-only return contract so the orchestrator can't be fooled by a worker that returns `green #<M>` before CI has actually completed. Closes [#56](https://github.com/mattsears18/claude-plugins/issues/56).

- **Stricter `green` semantics in `plugins/shipyard/agents/issue-worker.md`.** The fix-checks-only mode return-value spec now spells out unambiguously that `green #<M>` means "a full CI run completed and passed AFTER your final push" — not "pushed and queued," not "rebased onto a commit that should fix this," not "the failure looked transient so I optimistically declared victory." The contract is "the `statusCheckRollup` is fully `SUCCESS` / `SKIPPED` / `NEUTRAL` at the moment you return." A new worker-side `Don't` entry reinforces that hypotheses about a fix landing don't satisfy the contract — only an observed-green rollup does. The standard fix-loop's `gh pr checks <M> --watch --interval 30` step already drives this; the rewrite is for the ad-hoc dispatch case where a custom prompt drops the watch step.
- **Orchestrator-side trust-but-verify in `plugins/shipyard/commands/do-work.md` step A reconcile.** Every `green #<M>` and `noop: already green #<M>` return now triggers a one-call `gh pr view <M> --json statusCheckRollup,mergeStateStatus` spot-check. The orchestrator walks the rollup and downgrades silently:
  - any `PENDING` / `IN_PROGRESS` / `QUEUED` entry → return becomes `pending`, `<M>` is added to `session_prs` for the end-of-session drain to watch, advisory line logged: `[fix-checks-verify] downgraded #<M> green→pending: ... drain will reconcile.`
  - any `FAILURE` / `ERROR` / `TIMED_OUT` entry → return becomes `failing`, `<M>` is pushed onto `failed_prs` (deduped) for the next dispatch cycle to pick up, advisory logged: `[fix-checks-verify] downgraded #<M> green→failing: ... re-queued for fix-checks.`
- **New `Don't` entry in `commands/do-work.md`.** "Don't accept a `green #<M>` return from a fix-checks-only worker without verifying the rollup" — documents the failure mode and the spot-check as its defense, with a back-reference to the motivating bug.
- **Hardened fix-checks dispatch prompt template.** The prompt the orchestrator sends to fix-checks-only workers (in the dispatch-rules section) now restates the `green` contract inline so workers see the strict definition even before reading the agent spec: "**`green #<M>` means a full CI run completed and passed AFTER your final push — not 'pushed and queued.'**"

Discovered after a `mattsears18/lightwork` ad-hoc dispatch on PR #824: the orchestrator authored a one-off "rebase the PR onto current main, lift `ci-blocked`, the recent bundle-shrink work probably fixes the webserver hang" prompt that explicitly told the agent to skip `--watch`. The agent rebased, lifted, and returned `green` optimistically. The orchestrator trusted the return, the session ended, and CI subsequently hung all 3 E2E shards for the full 20-min timeout — exactly the same failure mode that triggered the original `ci-blocked` label. PR #824 sat red and unlabeled until the next session's step 5 picked it up. The trust-but-verify spot-check makes that path impossible: even if a future ad-hoc prompt drops `--watch`, the orchestrator catches the discrepancy and downgrades to `pending` / `failing` automatically. Issue [#56](https://github.com/mattsears18/claude-plugins/issues/56) also proposed a canonical `lift-ci-blocked` dispatch mode and a history-aware retry counter; those remain valuable but are intentionally out of scope for this release — split into follow-up issues so this fix lands minimally.

### 1.2.7 — 2026-05-20

Hardens worktree teardown so the orchestrator never reaps a still-running agent's worktree, and gives the issue-worker template an explicit escape hatch if it ever happens anyway. Closes [#64](https://github.com/mattsears18/claude-plugins/issues/64).

- **End-of-session cleanup step 3 now liveness-checks before reaping.** Previously the shutdown reap was unconditional — "at shutdown every dispatched agent is done, regardless of lock state." That assumption was load-bearing on correct termination logic, and the premature-termination bug (#57) demonstrated the failure mode: cleanup fired while a dispatched fix-checks agent was mid-rebase, its worktree got force-removed, and the agent returned later with `blocked: assigned worktree ... was cleaned up mid-session`. The exact return string is load-bearing for step A's reconcile parsing.

The plugin started as an audit suite but grew into a general-purpose autonomous engineering loop: audits surface work, users feed work in, an orchestrator refines / dispatches / fixes / ships, and a fleet of specialized agents do hands-on work in parallel worktrees. The old name no longer fit. `shipyard` captures the mental model — many specialists working in parallel on different vessels at every stage of their lifecycle, coordinated by a foreman.

Migration: anything referencing `app-audits:*` in scripts, settings, or `subagent_type` dispatches must flip to `shipyard:*`. No alias layer ships; this is a clean cutover.

Other changes in this release:

- `plugin.json` description + keywords rewritten to reflect the broader scope (orchestrator, autonomous-agents, backlog-burndown alongside the existing audit-flavored keywords).
- `marketplace.json` entry: `category` flipped from `quality` → `engineering`; `source` repointed to `./plugins/shipyard`.
- Statusline cache TTL env var renamed: `APP_AUDITS_STATUSLINE_CACHE_TTL` → `SHIPYARD_STATUSLINE_CACHE_TTL`.
- `enforce-worktree-isolation` hook now matches `subagent_type: "shipyard:issue-worker"` in incoming tool calls (was `app-audits:issue-worker`).
- README, plugin tree, and all internal cross-references updated atomically.

### 0.15.x and earlier

Released under the `app-audits` name. See [git log](https://github.com/mattsears18/claude-plugins/commits/main/plugins/shipyard) for the full history.
