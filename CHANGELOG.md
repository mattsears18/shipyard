## shipyard

### 1.8.62 — 2026-06-09

Surfaces non-clearable `blocked:agent-hard` refuses as a `/my-turn` human-review item (#500). Before this change, `/my-turn` Pass B only surfaced a `blocked:agent-hard` issue when it was *auto-clearable* — i.e. when every `Blocked by #N` reference in the body was already CLOSED/MERGED (a P2 "housekeeping: remove the stale label" item). The common case — a genuine security / scope / prompt-injection *refuse* that `steady-state.md`'s bail-classification table stamps `blocked:agent-hard` precisely because it "genuinely needs human review" — has no `Blocked by` references at all (or has an open one), so it never surfaced. The result: `/do-work` hard-blocks the issue out of dispatch across sessions while `/my-turn` stays silent, and real refuses stack up with no handoff. The fix splits the signal by clearability: a non-clearable hard-block (≥1 referenced blocker still open, OR zero `Blocked by` refs) now ranks **P1 — decisions** with the action "review the refuse reason in the comment; clear, re-scope, or close"; the clearable subset stays **P2 — housekeeping**.

- **`plugins/shipyard/commands/my-turn.md`** — Pass B's `blocked:agent-hard` bullet split into non-clearable (P1) and clearable (P2) signals with the `Blocked by #N` reference-state decision rule (zero refs ⇒ refuse ⇒ non-clearable); the P1 — decisions and P2 — housekeeping ranking tiers updated to carry the two cases; Rendering-rules action-verb examples extended with both the non-clearable ("review the refuse reason…") and clearable ("remove the blocked:agent-hard label…") actions.
- **`plugins/shipyard/scripts/tests/my-turn.test.sh`** — four new assertions (non-clearable split documented, clearable-vs-non-clearable distinction, the `Blocked by` keying, and the `#500` citation), bringing the suite to 29 assertions.
- **`plugins/shipyard/.claude-plugin/plugin.json`** — version bump to 1.8.62.

### 1.8.60 — 2026-06-08

Applies a `needs-decomposition` surfacing label when a `/do-work` scope agent confirms an issue is non-shippable as a single PR (the "epic" case), closing the human-handoff gap (#498). Previously, when the scope pre-flight reached `confirmed-non-shippable-as-single-PR` it posted a diagnosis comment but applied *no label*, so the issue fell through the cracks of both automated loops: `/do-work` re-scoped it every single session (a scope agent burning tokens to re-derive the same "too big" conclusion), and `/my-turn` — being entirely label-driven — never surfaced it as a human decision. The intended diagnosis-comment handoff never reached the human queue. The fix adds a dedicated `needs-decomposition` label (semantically distinct from `needs-design` / `needs-triage`), applies it only for the `confirmed-non-shippable-as-single-PR` defer class (the other four defer classes are not decomposition handoffs and keep the no-label behavior), and adds it to `/do-work`'s dispatch-exclusion set so the same epic stops being re-scoped. The drain-phase scope-agent re-validation removes the label again if a fresh pass finds the issue ready (a slicing-miss defer can still recover). The companion `/my-turn` surfacing side is tracked separately in #499.

- **`plugins/shipyard/commands/do-work/setup.md`** — registers `needs-decomposition` in both idempotent label-ensure blocks (the step 0.7 background group, comment bumped from "12" to "13 idempotent labels", and the canonical step 3a block plus its prose intro); the step 6 Deferred recording path now applies the label via `gh issue edit --add-label needs-decomposition` gated on `defer_reason_class == "confirmed-non-shippable-as-single-PR"`, replacing the old "Do not add a label" instruction (other classes keep the no-label behavior); adds `needs-decomposition` to the step 4 dispatch-exclusion set and to the "enumerate each literal label name" rationale note.
- **`plugins/shipyard/commands/do-work/drain.md`** — the 5.b scope-agent re-validation ready-shape branch now removes `needs-decomposition` (gated on the original `confirmed-non-shippable-as-single-PR` class) when a re-validated defer flips to ready, with a `(removed needs-decomposition)` log suffix, so a slicing-miss epic isn't left hidden behind the new exclusion label.
- **`plugins/shipyard/commands/do-work.md`** — the `deferred_issues` orchestrator-state note documents the new label application for the unsliceable class and the drain-phase removal.

### 1.8.59 — 2026-06-08

Hardens every worktree-reap block in the steady-state loop against a cwd-leak hazard distinct from the derivation-leak fixed in #452/#477 (#497). When the harness leaks the orchestrator's Bash-tool cwd into a just-returned agent's `agent-*` isolation worktree, and the orchestrator then reaps *that* worktree on the reconcile turn, `git worktree remove --force` deletes the directory the shell is standing in — and every subsequent bare git command in the same block (the `git worktree prune`, any follow-up `fetch`/`log`) dies with `fatal: Unable to read current working directory`. The reap silently half-fails, and on the reconcile turn it can skip the post-merge CI watch on the exact `merged-direct-ungated` path where that watch matters most (repro: session `do-work-20260608T204506Z-36559` reconciling PR #496). The spec's old recovery (`cd "$(git rev-parse --show-toplevel)"`) could not rescue the block — git resolves its own deleted cwd before reading anything, so the rev-parse fails first; empirical testing also showed `git worktree list --porcelain` (the idiom the issue's suggested fix assumed would survive) fails the same way. The fix anchors cwd to a stable directory derived *cwd-independently* via the #477 porcelain idiom (orchestrator worktree first, primary as fallback) and `cd`s to it **before** the reap, while cwd is still valid — so the remove/prune never run with cwd inside the doomed directory.

- **`plugins/shipyard/commands/do-work/steady-state.md`** — added a `STABLE_DIR` porcelain-anchor preamble (`cd "${STABLE_DIR:-/}"`) to all four reap blocks: A.0.5 crash reap, the A.1 `shipped`-immediate reap (#282), step B's per-completion reap (#334), and step C 2d's pre-dispatch reap (#368). The A.1 / step B / step C blocks additionally re-derive their `.git/worktrees` walk root from the porcelain-derived primary instead of the leaked cwd. A new "cwd-anchor-before-reap invariant" prose section documents the hazard, the cwd-independence requirement, and why `git rev-parse --show-toplevel` cannot recover a deleted cwd.
- **`plugins/shipyard/scripts/tests/reconcile-reap-cwd-anchor.test.sh`** — new 10-assertion regression guard: spec assertions that all four reap blocks carry the porcelain anchor and name #497, plus a runtime proof (negative control: the naive `cd $(git rev-parse --show-toplevel)` recovery fails on a deleted cwd; positive: anchor-before-reap lets `git worktree prune` and follow-up git commands survive).
- **`plugins/shipyard/.claude-plugin/plugin.json`** — version bump to 1.8.59.

### 1.8.58 — 2026-06-08

Extends the A.0.5 pre-reap crash-recovery (introduced in 1.8.57 / #493) to also salvage UNCOMMITTED working-tree edits, not just committed-but-unpushed commits (#495). The 1.8.57 trigger (`rev-list --count origin/<default>..HEAD > 0`) fires only when the worker managed to commit before crashing; a worker that stalled *before* committing — e.g. killed by the stream watchdog mid-version-bump, or hung in a pre-commit hook — has `rev-list --count == 0` and the previous recovery skipped, discarding the edits. This session's own #493 implementation was dogfooded: the worker stalled after writing ~80 lines of code + a 20-assertion test (both uncommitted), and the orchestrator had to go off-spec to hand-recover the files. The fix adds a `elif git status --porcelain` branch: when `rev-list` finds nothing but the working tree is dirty, the orchestrator stages all changes and commits with `--no-verify` (the pre-commit gate may be exactly what hung the worker; CI is the real safety net), then falls through to the existing push → PR-create → arm-auto-merge path from #493. The recovery is gated to `kind == "issue"` dispatches, is fully fire-and-forget, and always proceeds to the reap afterward.

- **`plugins/shipyard/commands/do-work/steady-state.md`** — "Pre-reap recovery check" header and issue link updated to include #495; step 1 of "Recovery semantics" extended with the dirty-working-tree case; the code block gains a `elif [ -n "$(git status --porcelain)" ]` branch that runs `git add -A` + `git commit --no-verify`, sets `push_ok`, and falls through; the shared push+PR-create+auto-merge block is extracted outside the `if/elif` and guarded with `push_ok="${push_ok:-false}"` so the clean-worktree (no recovery) case can't trip a `set -u` unbound-variable error.
- **`plugins/shipyard/scripts/tests/crash-recovery-A05.test.sh`** — extended from 20 to 29 assertions: the new #495 traceability check, `status --porcelain` detection, `commit --no-verify`, `git add -A`, the dirty-worktree log marker, the `elif` → shared-block document ordering, the `push_ok` default guard, the `slot_kind == "issue"` scope coverage, and the `#495` prose reference.

### 1.8.57 — 2026-06-08

Adds a pre-reap recovery check to the `/shipyard:do-work` orchestrator's A.0.5 crash handler so a worker that crashes *after* committing locally but *before* pushing no longer has its work discarded (#493). The previous spec reaped a crashed worker's `agent-*` worktree and dropped its `do-work/issue-<N>` branch unconditionally, then re-enqueued the issue for a from-scratch redo — throwing away a hook-validated commit (typecheck/lint/format/secrets already passed) and burning another full dispatch plus the user's wall-clock. The new check inspects the crashed branch for commits ahead of `origin/<default>` *before* the reap: when found, it pushes the branch, opens (or reuses) a PR with the standard `Closes #N` + `--label shipyard` template, arms auto-merge, and appends the recovered PR to `session_prs` so drain and the end-of-session summary see it — converting a mid-session network blip from "lose N committed PRs + redo from scratch" into "push + open PR for the work already done." The recovery is gated to issue-work dispatches (slot `kind == "issue"`), is fully fire-and-forget (a failed push or PR-create logs but never aborts the reconcile turn), and always proceeds to the reap afterward. This very session dogfooded the gap: the #493 worker stalled at the version-bump step with the implementation uncommitted, and the work was hand-recovered to ship this PR.

- **`plugins/shipyard/commands/do-work/steady-state.md`** — A.0.5 gains a "Pre-reap recovery check" prose section (rev-list signal, push → PR-create → arm auto-merge → `session_prs` append ordering, issue-work-only scope, fire-and-forget posture) plus the corresponding recovery code block ahead of the existing reap; the scope guard reads the slot's documented `kind`/`target` fields (`kind == "issue"`); fire-and-forget and audit-log-shape notes extended to cover the recovery steps and the `[reconcile-A.0.5-recovery]` stdout log prefix.

### 1.8.56 — 2026-06-06

### 1.8.55 — 2026-06-06

Adds a step-0 cwd fail-fast guard to the shared worker preamble so a misrouted `isolation: "worktree"` dispatch bails immediately instead of burning its entire run and dying at the final commit (#486). A Claude Code *harness*-level misroute can land an `isolation: "worktree"` dispatch with its process cwd pinned to the **primary checkout root** rather than the created worktree; on any repo carrying the global `enforce-worktree.sh` PreToolUse hook (the exact environment shipyard's worktree-isolation contract assumes), every git-mutating command then targets the user's primary checkout and the hook hard-blocks the worker's final `git commit` — but only after the worker has implemented, validated, and staged the whole task (the issue's repro: ~94 min of Opus dying at the last step, with no in-session path to recover the staged work). The harness owns the cwd, so AC items (1) and (2) of #486 — forcing the dispatch/orchestrator cwd to the correct worktree — are **out of shipyard's control**; the in-repo mitigation is AC items (3) and (4): a pre-flight assertion that detects a primary-checkout cwd via git's own truth (`git rev-parse --git-dir` == `--git-common-dir`, which holds only for a primary checkout and is false for every linked worktree) and returns `blocked: dispatch-isolation cwd override is wrong` at step 0, plus a behavioral regression guard over a real primary-checkout + linked-worktree fixture. The guard runs in every worker mode and is a cheap no-op on the normal (correctly-pinned) path.

- **`plugins/shipyard/scripts/tests/worktree-cwd-fail-fast.test.sh`** — new behavioral regression suite (AC 4): builds a real primary checkout + a `git worktree add` linked worktree and asserts the detection predicate classifies primary (incl. nested subdir) as `primary` and the linked worktree (incl. nested subdir) as `worktree`, path-independently; also pins the SKILL.md wiring of the predicate.

### 1.8.54 — 2026-06-03

Fixes the `guard-primary-checkout` hook to fail-open when `cwd` is absent from the hook payload, restoring green CI on `main` (PR #484). The hook's fail-open contract (issue #482: "fail-open on any uncertainty") requires exit 0 when the input payload is missing fields needed to determine whether we're in the primary checkout. The `missing tool_input/cwd → allowed` regression test shipped in 1.8.53 correctly captured this contract, but the hook itself was falling back to `${PWD:-}` when `cwd` was absent from the JSON payload — if the hook process's `$PWD` happened to be the primary checkout (e.g. when the CI runner's shell cwd was not inside a linked worktree), the guard would fire and exit 2 even though the payload lacked a `cwd` field entirely. The fix removes the `$PWD` fallback and exits 0 immediately when `cwd` is empty after JSON extraction, consistent with the other fail-open cases (`not-a-repo`, `malformed JSON`, `empty object`). 25/26 tests passed before; all 26/26 pass after.

- **`plugins/shipyard/hooks/guard-primary-checkout.sh`** — removed `cwd="${PWD:-}"` fallback; `[[ -z "$cwd" ]] && exit 0` now fires immediately when the payload omits `cwd`, matching the fail-open contract.

### 1.8.53 — 2026-06-02

Adds an opt-in primary-checkout guard offered during `/shipyard:init` (#482). `/do-work` isolates *background* workers in their own worktrees and the native `worktree.bgIsolation` setting isolates *background* sessions, but nothing protected a user's *interactive* sessions from colliding in the repo's primary checkout — two sessions sharing one working tree can switch the branch / dirty the index out from under each other (the issue's real-world repro: a second session switched the shared checkout to a feature branch mid-task while the first believed it was on a clean `main`). The guard is a `PreToolUse` hook that fires when an `Edit`/`Write`/`git commit` runs in the primary checkout rather than a linked worktree, nudging each editing session into its own `git worktree`. It ships **off by default** and is wired into the user's own `settings.json` (never the plugin's always-on `hooks.json`), with three modes (`off`/`warn`/`block`) and a scope choice (repo vs global). Per the issue's carrot+stick refinement, `/shipyard:init` offers the hook (stick) alongside a paired CLAUDE.md "enter a worktree first" rule (carrot) and documents the `ExitWorktree` cleanup lifecycle (tied to merge state, not "worker finished"); the block message leads with the `EnterWorktree` tool rather than a bare `git worktree add`.

- **`plugins/shipyard/hooks/guard-primary-checkout.sh`** — new reference hook. Detects the primary checkout via git's own truth (`--absolute-git-dir` == `--git-common-dir`, both realpath-normalized so a `/tmp`↔`/private/tmp` symlink doesn't false-positive); reads `SHIPYARD_PRIMARY_GUARD` (`off`/`warn`/`block`) so one script serves all policies; scopes to `Edit`/`Write`/`MultiEdit`/`NotebookEdit` + `git commit` Bash calls; fails open on any uncertainty (not-a-repo / malformed input / detection error → allow). Not registered in `hooks.json` — init wires it into the user's `settings.json` via `${CLAUDE_PLUGIN_ROOT}`.
- **`plugins/shipyard/hooks/tests/guard-primary-checkout.test.sh`** — new regression suite (26 assertions) over a real git primary-checkout + linked-worktree fixture: block/warn/off modes, the linked-worktree allow path, the `git commit`-only Bash scoping, the `EnterWorktree`-first block message, and the fail-open cases.
- **`plugins/shipyard/commands/init.md`** — new interactive prompt + `--primary-guard`/`--primary-guard-scope` flags + step 9.5 (install logic: settings.json merge, paired CLAUDE.md rule, idempotent re-run), plus Related links to the issue and the hook script.

### 1.8.52 — 2026-06-02

Hardens the issue-worker's end-to-end close guarantee so a resolving PR actually closes its dispatched issue on merge (#481). The worker contract implicitly promises the issue gets closed when its PR lands, but nothing enforced that the PR used a GitHub **closing keyword** (`Closes`/`Fixes`/`Resolves #N`) rather than a bare reference (`Refs #N`, `Related to #N`, plain `#N`). A bare reference does NOT register a closing link, so the issue silently lingers OPEN forever after merge — the work ships, the issue stays open, and `/do-work` can re-pick an already-resolved issue (polluting the "zero matching issues remain" termination signal). The real-world repro: `lightwork#1581` merged with `Refs #1580`, leaving #1580 OPEN. Likely cause: a worker over-applied a repo-local "don't auto-close" convention (correct for *incidental* references) into "always use `Refs #N`" — but the resolving PR for a dispatched issue is the intended-close case and must use the keyword. The fix (a) makes the step-5 keyword requirement explicit about closing-keyword-vs-bare-reference and carves out the repo-local-convention exemption for the dispatched issue, (b) adds a new post-create verification step that asserts GitHub registered `#N` as a closing reference and auto-patches the PR body with `Closes #N` (then re-verifies, bailing to `needs-human-review` if the link still won't register), and (c) reinforces the shared worker-preamble's PR-creation contract with the same distinction.

- **`plugins/shipyard/agents/issue-worker/issue-work.md`** — step 5 gains an explicit closing-keyword-vs-bare-reference paragraph plus the repo-local "don't auto-close" exemption for the resolving PR; new § "5.8 Post-PR-create closing-link verification" asserts `closingIssuesReferences` contains `#N` and patches the body + re-verifies before arming auto-merge; a new "Don't" bullet pins the bare-reference prohibition.
- **`plugins/shipyard/skills/worker-preamble/SKILL.md`** — the PR-creation contract's closing-line one-liner is expanded to spell out the closing-keyword requirement, the bare-reference failure mode, and the repo-local-convention exemption, pointing at the new issue-work.md sections.

### 1.8.51 — 2026-06-01

Fixes the step 1.3 #465 admin-direct-merge detector (#479), which was silently suppressing its own warning on precisely the repos it targets — admin + an unprotected default branch (the shipyard repo itself). The required-status-checks read `gh api .../required_status_checks --jq '(.checks // []) | length' 2>/dev/null || echo 0` is not 404-safe: `gh api` does NOT apply `--jq` to error responses, so on a 404 (`Branch not protected` — exactly the zero-required-checks shape #465 warns about) it writes the raw error JSON to *stdout* and exits non-zero, and the `|| echo 0` fallback then *appends* `0` to that body rather than replacing it (`required_checks_count=[{"message":"Branch not protected",...}0]`). That non-numeric value never equals `"0"`, so shape-2 of the detector (`admin && required_checks_count == 0`) never matched. The fix adds a one-line numeric-shape normalize (`case "$required_checks_count" in ''|*[!0-9]*) required_checks_count=0 ;; esac`) right after the read — option 1 from the issue — collapsing any non-digit value (404 body, empty, transient failure) to `0`, which is the correct semantic (a 404 means zero required checks) and matches the warn-on-doubt direction the step-1.3 comment already documents. The trailing prose paragraph is corrected to describe the actual (now-working) behavior.

- **`plugins/shipyard/commands/do-work/setup.md`** — step 1.3's required-checks read gains the `case` numeric-shape normalize plus a comment explaining the `gh api` 404 append-not-replace trap; the closing paragraph's claim about the `|| echo 0` fallback is rewritten to reflect the fix.
- **`plugins/shipyard/scripts/tests/required-checks-404-normalize.test.sh`** — new regression suite (10 assertions): pins the normalize block's presence and ordering in setup.md, reproduces the pre-fix bug (bare `|| echo 0` on a 404 ≠ `"0"`), and proves the normalize collapses the 404 body / empty string to `0` while preserving genuine counts, so shape-2 fires on admin+404 and stays quiet when required checks exist. Full `*.test.sh` sweep passes; shellcheck clean.

### 1.5.29 — 2026-05-29

Closes [#374](https://github.com/mattsears18/shipyard/issues/374) (P2, bug) — the **end-of-session drain's 120-min wall-clock ceiling exited prematurely on healthy-but-slow merge trains**. The old ceiling fired on the wrong signal: a multi-PR session with full E2E sharding (3 shards × ~10-15 min), Lighthouse (~9 min), and release-please cascades (each merge re-DIRTY-ing siblings, then a 15-25 min rebase + re-CI round-trip) can legitimately take >2h to land even when nothing is wrong, and the cap cut those sessions off mid-flight — forcing the user to re-invoke `/do-work` to resume the drain (which also resets cost tracking). The fix replaces wall-clock with a **per-PR progress signal**: drain tracks each open PR's `headRefOid` and the timestamp it last moved (`head_unchanged_since`), and a PR counts as "settled" once its head has been unchanged for `ci.settled_minutes` (default 20) — CI grinding but the branch not being actively reworked. Drain exits the moment every session_pr is settled (merged/closed, `blocked:ci`, rebase-blocked, rate-limit-capped, or pending-head-quiet), with no fixed-poll no-progress window. A new `ci.max_drain_hours` (default 8h) survives as the belt-and-braces outer bound so a degenerate session (runaway CI loop, infinite rebase whose head moves every poll) still can't run forever. The per-poll status line now surfaces the progress clock per still-pending PR (`PR #N: head unchanged 12m / 20m settled-threshold`) so the operator can watch the signal that determines settle. Docs/spec + config-schema + test change — the drain loop is orchestrator-interpreted prose, so the behavior change lands in `drain.md` rather than a helper script. Companion to #373 (cross-session DIRTY adoption into `session_prs`, shipped earlier this session).
