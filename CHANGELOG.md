## shipyard

### 1.4.23 — 2026-05-24

Closes [#305](https://github.com/mattsears18/shipyard/issues/305) (P1) — `commands/init.md` step 5 told the assistant to call `shipyard-config.sh set repo.owner ... --repo` followed by `set repo.name ... --repo` as two separate writes, but `.repo` is the one place in `schemas/shipyard.config.schema.json` that declares a `required: ["owner", "name"]` pair. Each per-field `set` commits the intermediate state to disk and the validator refuses to leave `.repo` half-populated, so the first call fails with `schema validation failed: .repo: missing required field name` and `init` never gets a working config off the ground. Hit dogfooding `/shipyard:init` on `mattsears18/mattsears18.com` on 2026-05-24 against shipyard@1.4.17 — the helper script is correct (per-write validation is the load-bearing safety net for typos in `auto_merge.policy` etc.), it's the docs that conflict with the schema's required-pair shape. The smaller fix per the issue body's option 1 is to write nested required-pair objects as a single JSON-object `set` call (the suggested workaround `set repo '{"owner":"...","name":"..."}' --repo` — confirmed working in the repro). The "one `set` call per field" rule still applies for every independent field; the exception is documented inline with a pointer to #305 so a future maintainer reading the prose doesn't try to "fix" the inconsistency by reverting to per-field writes.

- **`plugins/shipyard/commands/init.md`** — step 5's example block reshaped: `set repo.owner` + `set repo.name` (two calls) collapsed into one `set repo '{owner, name}'` call built via `jq -nc --arg`. Surrounding prose updated from a flat "One `set` call per field so each write goes through schema validation" to "One `set` call per field … with one exception: nested objects whose schema declares a `required: [...]` pair (currently just `.repo`, which requires both `owner` and `name`) MUST be written in a single `set` call …" with the rationale (per-field writes commit intermediate state the validator rejects) and a #305 cross-reference. All other field writes (`auto_merge.policy`, `trust.authors`, `cost_tracking.enabled`, `concurrency.default`) keep the per-field shape unchanged.
- **`plugins/shipyard/.claude-plugin/plugin.json`** — version bump 1.4.22 → 1.4.23.

### 1.4.22 — 2026-05-24

Closes [#302](https://github.com/mattsears18/shipyard/issues/302) (P1) — the upstream complement to #298 (classified defers) and #299 (drain-time re-validation): the scope-agent prompt in `commands/do-work/setup.md` step 6 was still permissive enough that a scope agent could pick a valid `defer_reason_class` from the five-value enum and produce a `reason` that *sounded* mechanical without actually being grounded in evidence. Session experience on `mattsears18/lightwork` confirmed the gap — most enhancement issues there carry `labels: enhancement, web, ios, android, P2` with one-paragraph bodies, and scope agents reading those bodies routinely returned `deferred` with reasons like `"cross-platform — looks like a multi-PR migration"` and picked `confirmed-non-shippable-as-single-PR` as the class. The class was a valid enum value; the reason was speculative judgment. Each such defer silently dropped a sliceable issue from the session, and the #299 drain-time re-validation could only re-confirm the same speculative read (because the body hadn't changed) and burn a fresh scope-agent invocation doing it. The fix has two parts that compose with the existing #298 + #299 gates: (1) a tightened prompt that catalogs the four canonical speculative-judgment phrases by name (`"looks like a multi-PR migration"`, `"touches three platforms — likely complex"`, `"UX change — probably needs design review"`, `"body is vague"`) so the agent can pattern-match its own draft against them before returning; (2) a required `evidence_pointer` field on every deferred return, validated by the orchestrator against a per-class shape table — speculative pointers are rejected inline at scope time (the issue goes back onto `raw_backlog` for a fresh scope pass) instead of waiting for the drain-time re-validation safety net. The two-gate model means a session with no speculative defers sees no re-validations at all; speculative defers get caught inline at near-zero cost; defers with evidence that looked valid but turned out wrong on second look still get caught by #299's drain-time pass.

- **`plugins/shipyard/commands/do-work/setup.md`** — step 6's deferred shape grew a required `evidence_pointer` field with a per-class shape table (5 rows, one per `defer_reason_class` value) and a rejected-shapes catalog naming the four speculative-judgment phrases verbatim. The scoping-agent prompt instruction rewrite tightens "Return the deferred shape ONLY when there's no first phase that can ship independently" to "Return the deferred shape ONLY when you can cite SPECIFIC MECHANICAL EVIDENCE for the defer — a named open blocker issue (`Blocked by #N`), a missing dependency the worker can't reasonably install/lock/test/ship in the same PR, multi-service coordination that requires synchronized deploys, an external vendor change the worker can't move, or a referenced design artifact (Figma, RFC) that hasn't been imported into the codebase. Speculative reads of the body ... are NOT evidence and will be rejected by the orchestrator's per-class validator." The "Handling each returned entry → Deferred entries" block split into a Recording path (existing behavior, augmented to store `evidence_pointer`) and a new Rejection path (when `evidence_pointer` fails per-class validation: log the specific failure reason, comment on the issue, push it back onto `raw_backlog` preserving rank, do not increment `defers_this_turn`). The per-class validators are intentionally lightweight inline string/regex checks except `confirmed-blocker-still-open` which does a single `gh issue view <N> --json state -q .state` per cited blocker to confirm OPEN-ness.
- **`plugins/shipyard/commands/do-work/drain.md`** — step 5.b's scope-agent re-dispatch prompt now passes the original `evidence_pointer` to the fresh scope agent (alongside the existing `reason` / `defer_reason_class` / `would_be_dispatchable_as_phase_1_if` passthrough). When the fresh return is a deferred shape, the orchestrator runs the same per-class `evidence_pointer` validator from setup.md step 6 against the new pointer — a malformed re-validation promotes the entry to `raw_backlog` for a fresh scope pass (closing the loophole where a re-validation could re-confirm a defer with stale speculative evidence indefinitely). The pre-drain audit banner's spec-violation flag now covers missing/empty `evidence_pointer` the same way it covers missing `defer_reason_class` ([#298](https://github.com/mattsears18/shipyard/issues/298)).
- **`plugins/shipyard/commands/do-work/steady-state.md`** — step D's scope-refill sub-step note updated to reference the per-class `evidence_pointer` validator so the same rejection path applies to mid-session refill returns, not just initial pre-flight returns.
- **`plugins/shipyard/commands/do-work/cleanup-summary.md`** — `Deferred:` line spec gained a parallel spec-violation advisory for missing/empty `evidence_pointer` entries reaching the summary (treated the same way as missing `defer_reason_class`).
- **`plugins/shipyard/commands/do-work.md`** — `deferred_issues` orchestrator-state struct entry updated with the new required `evidence_pointer` field (per-class shape table is the canonical source in setup.md step 6 — do-work.md just notes it's required and references the table). Schema example updated to include `evidence_pointer: "Blocked by #1077"` on the existing #1075 sample entry.
- **`plugins/shipyard/commands/do-work-RATIONALE.md`** — new "Evidence-backed defers (issue #302)" section explains why the prompt + validator together are the right shape (defense-in-depth posture mirroring the body-vs-codebase rule in `issue-worker.md` step 2), the per-class shape catalog with worked examples, why upstream prompt-tightening is the right complement to #299's downstream drain-time safety net (upstream cuts speculative defers off at near-zero cost; downstream catches the survivors), and why drain-time re-validation now also runs the per-class validator on the fresh return (closing the loophole where stale speculative evidence could be re-confirmed indefinitely).
- **`plugins/shipyard/.claude-plugin/plugin.json`** — version bump 1.4.21 → 1.4.22.

### 1.4.21 — 2026-05-24

Closes [#301](https://github.com/mattsears18/shipyard/issues/301) (P1) — a silent backlog-shrink in `commands/do-work/setup.md` step 4 (and mirrored in `agents/issue-worker/issue-work.md` step 0) where three substring-search `gh pr list --search 'in:body "Closes #<N>"'` (plus `Fixes`, `Resolves`) queries per candidate issue were used as a belt-and-suspenders against the `-linked:pr` qualifier — but false-positived on release PRs whose CHANGELOG bodies itemized closed-by-this-release issues as `Closes #N` lines. Every itemization line silently suppressed the referenced issue from the workable queue, even though the PR wasn't actually closing those issues on merge. The repro on `mattsears18/lightwork`'s open release PR #1148 (`chore(release): 2.29.0`) showed `Closes #884`, `Closes #885`, `Closes #1051`, etc. — each one a CHANGELOG manifest entry, each one a silent suppression. Other repro shapes (meta-issue / epic PRs, comment-style PRs quoting older issues) hit the same pattern. The fix replaces the substring search with `closingIssuesReferences` — GitHub's canonical "this PR auto-closes that issue" GraphQL field, which matches exactly the issues that will auto-close on merge with no false positives on CHANGELOG manifests. The pattern is already established in `steady-state.md:316` for the mid-session re-evaluation sweep; this PR extends it to the setup-time and worker-pre-flight filters.

- **`plugins/shipyard/scripts/gh-batch.sh`** — `pr-status` projection extended to include `closingIssuesReferences(first: 100) { nodes { number } }` and the reducer flattens it to a `closingIssueNumbers: [N, ...]` array per PR. The `first: 100` upper bound is a sanity cap (no real PR closes more than ~10 issues); a future paginated-fetch issue would surface if any real PR ever exceeds it. Composes with `gh-cached.sh` like every other `pr-status` projection (10s TTL band) — setup-time and drain-time call sites can share the cache key.
- **`plugins/shipyard/scripts/tests/gh-batch.test.sh`** — 5 new assertions: the fake `gh` emitter now produces synthetic `closingIssuesReferences` (PR N closes issue N+1000 by default; magic number 888 → empty array for the no-closing-keyword case; magic number 777 → two closes for the multi-issue case), a new test block exercises the projection (the single-close case `142 → [1142]`, the empty case `888 → []` that is the exact shape that fixes the false-positive, and the multi-close case `777 → [1777, 2777]`), an assertion confirms existing fields aren't displaced, and a query-body shape assertion confirms `closingIssuesReferences` is requested. Suite grew from 50 → 55 passing assertions.
- **`plugins/shipyard/commands/do-work/setup.md`** — step 4's client-side filter replaces the three substring-search calls with a single `gh-batch.sh pr-status` invocation that builds a `closed-by-open-pr` set once per setup pass from open PRs' `closingIssueNumbers`. Membership check against that set replaces the per-candidate triple-search. New paragraph documents why the substring form was removed (CHANGELOG-manifest false positives), cites [#301](https://github.com/mattsears18/shipyard/issues/301), and notes the set is cached for the duration of step 4 (open PRs don't change between filter passes within one setup invocation).
- **`plugins/shipyard/agents/issue-worker/issue-work.md`** — step 0's open-PR cross-check replaces the same three substring searches with a single `gh pr list --json number,closingIssuesReferences --jq` projection that selects only PRs whose `closingIssuesReferences[].number` matches the candidate issue. Comment cites the same #301 rationale so the worker-side and orchestrator-side filters share the same justification.
- **`plugins/shipyard/.claude-plugin/plugin.json`** — version bump 1.4.20 → 1.4.21.

### 1.4.19 — 2026-05-24

Closes [#298](https://github.com/mattsears18/shipyard/issues/298) (P1) — a deeper failure mode behind the same family as #246 and #299: the orchestrator self-rationalizes defers and terminates while the backlog still holds actionable work, because the scope-agent prompt biases toward deferring instead of slicing multi-phase issues. Session `dowork-20260524T112421Z-63727` (+ resume) against `mattsears18/lightwork` entered drain three separate times while 24+ workable issues remained; each time the user had to intervene (*"why do you keep ending these do-work sessions early? there are still 24 issues in the backlog."*), and each re-evaluation pass successfully sliced 7 PRs out of issues the initial pass had deferred with reasons like "body explicitly recommends per-phase rollout". The #246 + #299 pre-drain re-validation pass added the right structural gate (the re-validation now fires whenever `deferred_issues` is non-empty), but when it fires it merely re-confirms the original defer because the body really IS multi-phase — re-validation as a gate against bad provenance worked, but re-validation as a way to find work in a backlog of legitimately-multi-phase issues didn't. The fix flips the scope-agent prompt's bias from "default to ready, defer when in doubt" to "default to slicing, defer only when no first phase can ship independently", adds a `phase_1_scope` field to the ready shape that flows through to the worker prompt so the dispatched worker knows what's in and out of the phase-1 envelope, and locks every deferred entry into a strict five-value `defer_reason_class` enum (`external-dependency`, `human-decision-required`, `untrusted-author`, `confirmed-blocker-still-open`, `confirmed-non-shippable-as-single-PR`) so plausible-sounding-prose defers can no longer slip past the audit.

Before this release, the orchestrator's drain phase (re-snapshotting `D_dirty` / `R_new` / `P_settled` every 60s for 3–8 session PRs) and the [setup-phase `blocker_state` cache](https://github.com/mattsears18/claude-plugins/blob/main/plugins/shipyard/commands/do-work/setup.md#08-blocker_state-cache-default-on) cold-start path both fired one `gh pr view <M>` or `gh issue view <N>` call per artifact. For N=8 PRs that's 8 network round-trips, 8 tool-result blocks in the agent's context, and 8x repeated query-framing tokens — when one GraphQL query with 8 aliased sub-queries returns identical data in a single response. - **`plugins/shipyard/scripts/gh-batch.sh`** — two subcommands. `pr-status --repo <owner/name> --numbers "<n> <n> ..."` emits a JSON object keyed by PR number string → `{number, state, mergeable, mergeStateStatus, statusCheckRollupState, headRefName, headRefOid}` (same projection a `gh pr view <M> --json ...` would return for those fields). `issue-state --repo <owner/name> --numbers "<n> <n> ..."` emits a parallel shape for issues — `{number, state, labels: [string, ...]}` with the labels array flattened from the GraphQL `nodes` shape so callers don't have to re-flatten. `--numbers` accepts space- or comma-separated integers; non-numeric tokens fail loudly (exit 64) as defense in depth against any caller injecting unvalidated input into the GraphQL body. Missing artifacts (deleted PR, transferred issue, never existed) resolve to a null GraphQL alias and drop from the output — caller treats a missing key as "not trackable." Failure of any chunk fails the whole batch with exit 2; no partial output is emitted so callers retry the whole batch rather than racing partial state.

Adds `plugins/shipyard/scripts/gh-cached.sh` — a session-scoped cache wrapper around the `gh` CLI so the orchestrator stops re-fetching the same data across phase transitions. Closes [#160](https://github.com/mattsears18/claude-plugins/issues/160) (Phase 3 of the perf umbrella [#152](https://github.com/mattsears18/claude-plugins/issues/152)). Lower-risk companion to [#163](https://github.com/mattsears18/claude-plugins/issues/163)'s cost ledger and the earlier do-work file split — small, well-scoped helper that opt-in callers wire in front of the hot `gh` calls.

Before this release, the orchestrator re-queried the same GitHub state on every phase boundary: `gh issue list` during setup, again on the dispatch-loop's lightweight backlog re-check, again on the periodic refresh; `gh pr list` at the start of dispatch, in drain, and in the end-of-session summary; `gh pr view <M> --json statusCheckRollup` on the green spot-check and again on drain reconcile. Within a 5–15 minute session most of those answers haven't changed — but they all hit the API anyway, burning rate-limit quota and adding measurable wall-clock latency to every phase transition. The new wrapper caches the stdout of read-only `gh` calls keyed by their full argv to `$SHIPYARD_HOME/cache/<session-id>/<sha256-of-argv>`, with a caller-supplied TTL, so the redundant re-fetches return from disk instead of the API.

- **`plugins/shipyard/scripts/gh-cached.sh`** — four subcommands: `run --session-id <id> --ttl <seconds> -- <gh-args...>` (cache-then-call wrapper; cache hit emits cached stdout + exit 0, miss runs `gh` and caches only on a zero-exit), `invalidate --session-id <id> [--pattern <substring>]` (drop cache files after a state-changing call; broad flush by default, optional pattern surface for targeted invalidation), `stats --session-id <id>` (emits `{"hits": N, "misses": N, "invalidations": N, "bytes": N}` for observability + perf measurement), and `cleanup --session-id <id>` (idempotent end-of-session reap of the cache directory). Atomic writes mirror the `session-state.sh` pattern — tmp-in-same-dir + `mv -f` rename, so a crash mid-write leaves the previous file intact rather than corrupting the cache. Failed `gh` calls (non-zero exit) are NOT cached — a transient 5xx or rate-limit error must retry naturally on the next call, never serve stale "success" data. Stderr is passed through live (warnings and rate-limit advisories still surface). `SHIPYARD_GH_CACHE_DISABLED=1` bypasses the cache entirely for debugging "is the cache hiding a real change?" without uninstalling.
- **TTL bands documented per call-site** in `commands/do-work/setup.md`'s new `0.9 gh-cached.sh wrapper (opt-in per call-site)` section. Suggested TTLs: 60s for `gh issue list` (backlog), 30s for `gh pr list` (in-flight), 10s for `gh pr view <N> --json statusCheckRollup` (CI churns fast), 600s for `gh label list` (rarely changes), 3600s for `gh repo view --json defaultBranchRef` and the collaborators query (session-stable). The wrapper is intentionally **opt-in per call-site** — the spec doesn't require every `gh` call to go through it; use it for the high-volume queries the orchestrator re-runs across phases.
- **Invalidation policy documented.** Two flavors: **conservative** (broad flush via `gh-cached.sh invalidate --session-id <id>` after any state-changing call shipyard itself makes — issue close, PR create, label edit; burns one extra round of cold reads on the next refresh but never serves stale data post-write), and **targeted** (`--pattern <sha-prefix>` when the caller knows which cached reads depend on the modified artifact; surface is intentionally narrow). The conservative policy is the default — the cost is small compared to the savings on the hot read paths.
- **Steady-state hot-path note added.** `commands/do-work/steady-state.md`'s "Lightweight backlog re-check (every dispatch)" block now points at the wrapper with `--ttl 60` as the recommended shape — that path fires every dispatch turn and is the single largest source of redundant `gh issue list` traffic.
- **End-of-session cleanup wiring.** New step 7.5 in `commands/do-work/cleanup-summary.md` reaps the cache directory between the cost-history flush (step 7) and the session-state cleanup (step 8) — `gh-cached.sh cleanup --session-id <id>`, idempotent, same don't-gate-exit posture as the surrounding cleanup chain.
- **`plugins/shipyard/scripts/tests/gh-cached.test.sh`** with 53 assertions covering: usage-error surface (no subcommand, unknown subcommand, missing flags, missing `--` separator, non-numeric TTL all exit 64); cache miss → cache file created + gh invoked once; cache hit → second call serves cached stdout + gh NOT re-invoked; different argv → different cache slot (order-sensitive keying); TTL=0 → always-stale (every call misses); backdated cache file (>TTL) → stale → miss; failed `gh` call → wrapper exit 2, stdout passed through, NO cache file written, subsequent calls also miss; `invalidate` (no pattern) drops every data file but preserves `_stats.json` so cumulative counters survive a flush; `invalidate` is idempotent on missing sessions; `stats` returns zeroed defaults for fresh sessions and accurate counts after activity; `cleanup` removes the dir and is idempotent on missing dirs; `SHIPYARD_GH_CACHE_DISABLED=1` invokes gh on every call and creates no cache dir; atomic-write hygiene (no `.tmp.<pid>` stragglers after successful writes). Picked up automatically by the existing `find plugins -type f -name '*.test.sh'` discovery in `.github/workflows/tests.yml`; shellcheck-clean.

Pure additive — no behavior change to any existing call-site (the wrapper is opt-in per site), no schema changes, no new runtime dependencies (`sha256sum` is standard on Linux, `shasum` on macOS; the wrapper detects which is available). The actual call-site swaps in `commands/do-work/*.md`, `commands/refine-issues.md`, `commands/my-turn.md`, and `agents/issue-worker/*.md` are deferred to follow-up commits once profiling identifies the highest-leverage paths — this PR ships the wrapper and the documentation surface so subsequent perf work has a substrate to read from.

- **Verb-first action.** Sentence-case imperative (`enable Email/Password in Firebase Console for <project>`, `set real values for placeholder secrets via firebase functions:secrets:set --project test`), not artifact-first (`PR #142 — "feat(shipyard): …"`). The user reads the verb and knows what to do; the issue number and URL are reference, not headline.
- **No tier headers.** Items are already ranked; their position in the list is the priority. Internal ranking still classifies into P0 / P1 / P2 (see [Ranking](./plugins/shipyard/commands/my-turn.md#ranking)) — the tiers just don't render. - **No per-item signal labels** (`needs-human-review`, `blocked:ci`, `awaiting your review`). - **No opening banner** (`HUMAN ACTIONS NEEDED — …`). Redundant — the user just typed the slash command. Cuts one line per invocation.
- **No closing prose summary.** Replaced by an optional one-line terse footer (e.g. `main: green · 1 blocked:ci PR`) surfaced only when there's something structural worth flagging — red main with no `fix-main-ci` PR, open `blocked:ci` PRs, or a clean state worth noting. Skip entirely otherwise.
- **Empty state** stays a single friendly one-liner (`Nothing on your plate — backlog is clean. Try /shipyard:audit to surface fresh work, or take a break.`) — no banner, no multi-line prose.
- **New `Don't add framing back to the output` entry** in the `Don't` section, pinned to this issue, so future maintainers and refactor agents have a clear regression boundary. The test invariants (`P0`, `P1`, `P2`, `URL`, `age`, `Don't`, etc.) still match — the test asserts presence of the *concepts*, not the rendering shape, and the `/my-turn` spec preserves every required input source, every tier classification, every protected non-goal.

Pure rendering change — no survey-pass changes, no `gh` call additions, no schema changes, no test invariant changes. Acceptance criteria from the issue body all met: single-line-per-item default, verb-first action, no tier headers, no closing prose summary, empty-state stays the friendly one-liner. First, a single-row "table" is degenerate — it pays the visual cost of table syntax (header, divider, pipes) without any of the alignment payoff. Second, even when multiple rows do render, markdown tables in many chat/terminal clients don't actually look table-like — column widths float to fit content (short labels like `#` collapse the count column, the Issues column dominates), the eye doesn't get the column-aligned scannable shape markdown tables are supposed to provide, and clients that show raw text just show raw pipes.

- **`commands/do-work.md` step 2 (backlog overview) — switched to two-mode rendering.** Count non-zero buckets at print time. `≥2` → fixed-width aligned text table with computed column widths (Bucket clamped to 30–60 chars, Count right-aligned with min width 5, Issues clamped to 30–80 chars), 3-space column separators, and a row of `─` (U+2500) characters under the header. No pipes, no markdown table syntax — reads the same whether the renderer processes markdown or not. `1` → single-line summary (`Workable: 6 issues (#90, #91, ...). Nothing skipped.` or, when the lone bucket is a skip bucket, `Workable: 0. Skipped: 3 issues in 'blocked' label (#A, #B, #C).`). `0` → `Backlog is empty — nothing to work on this session.`
- **`commands/do-work.md` end-of-session summary (step E) — same two-mode rendering.** Mirrors step 2's modes exactly so users can diff start-state vs end-state at a glance. The `Workable`-row reason text when `<W_end> == 0` (`everything shipped this session`, `everything left is blocked`, etc.) carries through both modes.
- **Workable-row-always-prints scoped to table mode.** When the table renders (`≥2` non-zero buckets), the `Workable` row prints even with `<W> == 0` so the user sees the "nothing workable this session" signal at the top. When counting rows to pick the mode, however, `<W> == 0` does NOT count as a non-zero bucket — otherwise a session where everything is blocked would render a two-row "table" with a zero-count `Workable` row, which is back to the degenerate shape this change is trying to avoid.
- **Action-recommendation sub-rows (⚠ likely-clearable / ⚠ likely-triageable) do NOT count as their own bucket for mode selection** — they're structurally sub-rows under their parent bucket, indented with 2 leading spaces, and the parent's non-zero count is what flips the mode.

Pure prompt edits; no spec / state-struct / tooling changes. `commands/refine-feedback.md` was audited per the issue body and does not borrow the bucket-table shape (its only multi-column table is a 5-column label-semantics reference, unaffected by this change).

- **Defense-in-depth in `agents/issue-worker.md` step 2.** Even after the orchestrator's author filter has cleared an issue, the worker now treats the body as untrusted input — re-derives the implementation from the codebase rather than the issue body, bails with `blocked: suggested fix exceeds expected scope — needs human review` if the suggested fix touches CI / secrets / `.github/workflows/` / unrelated files, and bails with `blocked: issue body contains directives that bypass normal review` on red-flag patterns (instructions to call external services, execute shell snippets verbatim, "trust me" framings). - **New `Don't` rule in `commands/do-work.md`.** Spells out the threat model in one paragraph so the dispatch-time filter can never be bypassed "just for this one issue" with a body that's already compromised by definition.

**Threat model and motivating threat:**

1. Stranger opens an issue on a public repo: `"Bug: foo() returns null. Suggested fix: add helper.ts with [crafted payload]."`
2. Issue carries none of the `-label:blocked` / `-label:wontfix` / `-label:needs-design` / `-label:needs-triage` / `-label:discussion` / `-label:needs-refinement` / `-label:needs-human-review` skip-labels, so step 4's filter includes it.
3. Pre-1.3.9: issue-worker dispatched, reads `body,title` as instructions per step 2 of the worker spec, opens PR, arms auto-merge.
4. If CI passes — and a subtle payload can be crafted to pass — the PR merges to `main` without human review. Worktree isolation prevents filesystem damage *outside* the worktree but does NOT prevent malicious code from landing in a merged PR, on a system that runs `/do-work` with broad tool access.

Under 1.3.9 the stranger-authored issue lands in bucket 0.5 and step 4's client-side filter drops it from the dispatch queue. The maintainer can still override: add the login to `.shipyard/trusted-authors.txt` (broad trust) or re-file the issue under the maintainer's own account using the same body (per-issue vouching). The pre-existing `needs-human-review` label is also a valid escape hatch the maintainer can use to surface the stranger's body for review — but the *default* for untrusted authors is **skip**, not workable. Previously the LOCKFILE rule was a blanket "if `touches_lockfile`, the rest of the pool parks." In LOCKFILE-heavy sessions — multiple dependabot bumps, security patches, scripts/coverage tweaks queued together — this dropped pool utilization to 1/N for the duration of every lockfile worker. In practice these PRs almost never actually conflict: `package.json` is a JSON object with disjoint top-level sections, and `package-lock.json` regenerates additively from the resolved tree post-merge. The all-park rule was overcautious.

- **`commands/do-work.md` dispatch rules — new section-aware lockfile rule.** `ready_issues` entries now carry `lockfile_sections: [...]` (e.g., `["overrides"]`, `["dependencies", "devDependencies"]`, `["scripts"]`, `["go.mod"]`) instead of the boolean `touches_lockfile` flag. Disjoint sections co-run normally — three workers claiming `overrides`, `dependencies`, and `scripts` respectively all dispatch in parallel. Treating them as claims would re-introduce the all-park behavior this change is undoing.
- **`idle_reason` valid set gains `all ready_issues collide with in_flight lockfile sections (<section>×<N>, ...)`** to name the *sections* in flight rather than the toucher's issue number — since multiple lockfile-touchers may be in flight simultaneously on disjoint sections, naming a single `#N` toucher would be ambiguous. - **`Don't` rule rewritten.** "Don't dispatch alongside a lockfile-toucher" → "Don't dispatch a candidate whose `lockfile_sections` overlaps an in-flight worker's `lockfile_sections`." Extends `/shipyard:do-work`'s setup-phase auto-clear sweep to also handle the `blocked` label, mirroring the `ci-blocked` pattern already in place. Closes [#73](https://github.com/mattsears18/claude-plugins/issues/73).

Previously the orchestrator auto-cleared stale `ci-blocked` labels on PRs but never touched stale `blocked` labels on issues. The asymmetry meant a `Blocked by #886, #887, #888` issue stayed labeled `blocked` forever even after all three blockers merged — and step 4's `-label:blocked` filter then silently hid it from every subsequent session's workable queue. ### 1.2.9 — 2026-05-20

Adds a drain-phase **fix-rebase** worker mode so the end-of-session drain can auto-rebase PRs whose only blocker is a stale base (`mergeStateStatus: DIRTY` with no failing checks). Closes [#61](https://github.com/mattsears18/claude-plugins/issues/61).

Previously the drain phase only dispatched fix-checks workers against newly-red PRs — DIRTY-but-green PRs sat untouched until the drain's no-progress termination criterion fired, leaving sessions to end with a tail of PRs needing manual rebase. After this change, the drain treats DIRTY PRs as actionable: a lightweight worker rebases onto current main, force-pushes with lease, and lets the merge train re-arm auto-merge naturally.

- **New `fix-rebase` mode in `plugins/shipyard/agents/issue-worker.md`.** Sits alongside `fix-checks-only` as a PR-targeted dispatch mode. The worker lands on the PR's head branch via the safe two-step (`git fetch origin <ref> && git switch <ref>`), pre-flights `mergeStateStatus` (bailing if no longer DIRTY or if hard failures appeared), runs `git rebase origin/<default-branch>`, applies a **trivial-or-bail conflict policy**, and `git push --force-with-lease`. The trivial-resolvable set is intentionally narrow: append-only conflicts in `CHANGELOG.md` / `CLAUDE.md` / `README.md` / shared docs, `ci.yml` shard-matrix appends, and lockfile entries that can be regenerated by re-running the package manager. Anything semantic (same function body / same imports / same JSON key with different values / file rename collisions / snapshot or fixture conflicts) hits `git rebase --abort` and returns `blocked rebase` immediately. The worker does NOT touch the PR title / body / labels, does NOT call `gh pr merge` manually (auto-merge was armed when the PR was opened and rebasing doesn't un-arm it), and does NOT `--watch` checks.
- **New drain-phase dispatch rule in `commands/do-work.md`.** Per-poll bookkeeping now tracks `D_dirty` (PRs in `session_prs` with `mergeStateStatus == "DIRTY"`, no hard-failure checks, not in flight, not yet in `rebase_blocked_prs`) alongside the existing `R_new` / `M_since_last` / `P_settled` sets. The drain dispatcher fills the pool with fix-checks workers first (red PRs are more urgent than stale-base PRs), then any remaining slots run fix-rebase workers — combined fix-checks + fix-rebase in-flight count stays under the session's `--concurrency` cap. Each fix-rebase is **one-shot per PR per session**: a `blocked rebase` return puts the PR into `rebase_blocked_prs` and the drain doesn't re-dispatch even if the PR stays DIRTY across subsequent polls. The drain's "settled" definition counts `rebase_blocked_prs` members as settled so a non-trivially-conflicted PR doesn't keep the drain alive indefinitely.
- **Drain status line gains visibility.** The per-poll `[drain] open=… newly_red=…` line now includes `dirty=<D_dirty>`, `rebase_blocked=<n>`, and a worker-count breakdown (`in_flight=<n>(fix-checks=<a>, fix-rebase=<b>)`) so the user can see at-a-glance how many PRs the rebase logic is working through.
- **End-of-session summary gains a `Drain-phase rebases` line** with the per-outcome breakdown (`<rebased_count> succeeded (#A, #B, ...)`, `<rebase_blocked_count> blocked (#C — <reason>, ...)`) — omitted entirely when both counts are zero so small sessions with no DIRTY churn stay terse. The `Drain phase` summary's `final session_prs state` partition also gains a `rebase-blocked: <r>` field alongside the existing `merged` / `ci-blocked` / `still pending` counters so the partition still sums to `|session_prs|`.
- **Open-PR drain query gains `headRefName,headRefOid`** so the fix-rebase prompt template has the branch identifiers it needs without a second `gh pr view` per dispatch. The fields are cheap to add to the existing rollup query.
- **New `Don't` rules.** Two on the orchestrator side ("don't dispatch fix-rebase outside the drain," "don't retry a `blocked rebase` PR within the same session"), two on the worker side ("don't resolve a non-trivial merge conflict in fix-rebase mode," "don't `gh pr merge` manually in fix-rebase mode") — both pairs cite #61 as the motivating bug.

Discovered after a `mattsears18/lightwork` session that ended with 6 of 11 open PRs in DIRTY state — #962, #961, #960, #958, #953, #824 all just needed `git fetch && git rebase origin/main && git push --force-with-lease` and would have auto-merged. The drain phase was capped at 15 min anyway under the old policy; by the time the cap fired, the merge train had advanced main far enough to invalidate most open PRs' bases, but the orchestrator had no mechanism to do anything about it. The new mode closes the gap without expanding scope into rewriting tests or touching PR descriptions.

### 1.2.8 — 2026-05-20

Hardens the fix-checks-only return contract so the orchestrator can't be fooled by a worker that returns `green #<M>` before CI has actually completed. Closes [#56](https://github.com/mattsears18/claude-plugins/issues/56).

- **Stricter `green` semantics in `plugins/shipyard/agents/issue-worker.md`.** The fix-checks-only mode return-value spec now spells out unambiguously that `green #<M>` means "a full CI run completed and passed AFTER your final push" — not "pushed and queued," not "rebased onto a commit that should fix this," not "the failure looked transient so I optimistically declared victory." The agent rebased, lifted, and returned `green` optimistically. PR #824 sat red and unlabeled until the next session's step 5 picked it up. - **Don't section L873 + L879 extended** to document the new lifecycle: applied by step A's fix-checks reconcile, removed only by step 3d's auto-clear sweep. Anything else flipping the label is foreign.

### 1.0.0 — 2026-05-19
