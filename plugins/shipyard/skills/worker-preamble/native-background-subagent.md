# Worker-preamble fragment — Native background-subagent auto-PR reconciliation

On-demand fragment of the `shipyard:worker-preamble` skill (see [`SKILL.md`](./SKILL.md)). This section self-declares a no-op on the happy path (a compliant worker's PR is already open and clean before the harness's native behavior would fire) and, post-[#791](https://github.com/mattsears18/shipyard/issues/791), narrows to non-`Workflow` dispatches — which is no `/do-work` mode. Load this only when hand-dispatching a mode shim via the `Agent` tool with `isolation: "worktree"` (e.g. `shipyard:verify-worker`), or when diagnosing a stray unlabeled draft PR left on a canonical `do-work/issue-<N>` branch. The per-mode specs point here by name (`shipyard:worker-preamble § "Native background-subagent auto-PR reconciliation"`).

## Native background-subagent auto-PR reconciliation ([#785](https://github.com/mattsears18/shipyard/issues/785))

As of Claude Code v2.1.198, `Agent`-tool subagents run **in the background by default**, and a background subagent that isolated its edits in a worktree (`isolation: "worktree"`) **auto-commits, pushes its own branch, and opens a draft pull request on its own** the moment it finishes — without asking, and independent of anything shipyard's own code does. This is a confirmed, current **harness-native** behavior (see [Claude Code's Agent View docs](https://code.claude.com/docs/en/agent-view)): *"A background session that isolated its code changes in a worktree also commits, pushes its own branch, and opens a draft pull request without stopping to ask ... it skips the pull request when you told it not to open one or the repository has no remote."* Per [Claude Code's sub-agents docs](https://code.claude.com/docs/en/sub-agents), this applies to **any** `Agent`-tool dispatch carrying `isolation: "worktree"`, regardless of whether the *top-level* session that spawned it is itself running in the background.

**Decision: the worker remains the sole, explicit owner of PR creation.** This section documents why that ownership doesn't race the native behavior in the normal case, and how the one residual gap is caught rather than silently producing a duplicate PR.

- **The happy path never races.** A compliant worker reaches its PR-creation step (issue-work [§5](../../agents/issue-worker/issue-work.md#5-commit--push--pr): `git commit` / `git push` / `gh pr create --label shipyard ...`) **before** its final turn ends — the "Return-contract discipline" rules already forbid ending a turn with uncommitted or un-PR'd work. By the time the harness's own end-of-subagent auto-commit/push/draft-PR logic would fire, the worktree is already clean and the branch already has an open, `shipyard`-labeled, closing-keyword-linked PR — the native behavior has nothing left to act on and is a no-op.
- **Tell it not to open one, explicitly.** The one documented opt-out the harness exposes is instructional ("it skips the pull request when you told it not to open one"), and every dispatch prompt loads this skill before doing any git work — so the instruction lives here: **do not rely on, wait for, or defer to the native background-completion behavior to open a pull request on your behalf.** The mode-specific PR-creation step is the only path that may call `gh pr create`; a PR opened any other way lacks the `--label shipyard` stamp and, for issue-work, the `Closes #<N>` keyword, breaking the hooks / orphan-triage / failing-PR-scan / end-of-session-summary machinery that keys off both.
- **The residual gap: a bail between push and PR-create.** If a worker pushes its branch and then hits a genuine blocker *before* `gh pr create` returns (a transient `gh` failure, a rate limit, an unrelated crash), the branch is left pushed with no PR — and the harness's native background-completion behavior, seeing an isolated worktree with a pushed-but-un-PR'd branch, may step in and open its own **draft** PR with no `--label shipyard` and no closing keyword. Issue-work's [step 0 pre-flight](../../agents/issue-worker/issue-work.md#0-pre-flight-confirm-the-issue-is-still-workable) cross-checks the **canonical head branch name** (`do-work/issue-<N>`) in addition to `closingIssuesReferences`, specifically so a re-dispatch against the same issue detects this kind of unlabeled, non-closing PR and bails rather than opening a second, competing one — see that step for the exact check.
- **Scope of the risk — narrow, as of [#791](https://github.com/mattsears18/shipyard/issues/791).** This race is specific to the `Agent`-tool subagent mechanism (`background` / `isolation` frontmatter). Every `mode:`-driven `/do-work` worker is now dispatched through the `Workflow` tool's `agent()` calls, which Claude Code's docs describe as a distinct mechanism with its own per-run budget — not a "subagent" in the `background`/`isolation`-frontmatter sense — so the native auto-commit/push/draft-PR behavior does not apply to them, and PR creation stays entirely orchestrator/worker-authored (see [`workflows/README.md`](../../workflows/README.md#worktree-isolation--the-callers-job)). The rules above therefore bind chiefly the remaining `Agent`+`isolation: "worktree"` dispatches — `shipyard:verify-worker`, and any hand dispatch of a mode shim — plus any repo running an older shipyard whose orchestrator still dispatches modes via the `Agent` tool. They cost nothing to follow either way, so follow them unconditionally.
