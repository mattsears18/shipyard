# /shipyard:do-work — Cold orchestrator-state structures

Eight of the orchestrator's mental data structures — each **consumed by exactly one other phase file**, not read on every turn by the dispatch loop itself. The thin entry [`commands/do-work.md`](../do-work.md) owns the [hot orchestrator-state struct list](../do-work.md#orchestrator-state) (the twelve structures + refresh tracker every turn's dispatch decision touches) and a pointer to the [session state file](./session-state-file.md); this file owns the long-tail structures that a single consumer phase file reads and writes — the flake-detection circuit breaker, the classifier-denial ledgers, the phantom-notification dedup set, the CI-minute / primary-checkout-leak counters, and the batch-dispatch version cursor. **Reference block — consulted by the one phase file that owns each structure, not executed top-to-bottom every turn** — so it lives in its own file to keep the thin entry under a byte budget a fresh session reads on every turn ([#808](https://github.com/mattsears18/shipyard/issues/808), mirrors the [dispatch-rules.md split #616](./dispatch-rules.md)).

## Cold orchestrator-state structures

- **`main_ci_fix_attempts`** ([#589](https://github.com/mattsears18/shipyard/issues/589)): session-local map `{ <earliest_red_workflow_name> → { attempts: <int>, last_pr: <#M|null>, last_sha: <string|null>, escalated: <bool> } }` implementing the fix-main-ci attempt-cap / flake-detection circuit breaker — the fix-main-ci analogue of the `blocked:ci` 3-attempt cap for fix-checks. Keyed by `earliest_red_workflow_name` (the stable cross-attempt signature — `earliest_red_sha` changes on every merge commit so it can't be the key, but a fix-main-ci PR that goes green on its own PR run yet leaves main's *merge-commit* red on the **same workflow** is exactly the flaky-test pass-on-PR/fail-on-merge pattern this counter detects). `attempts` is incremented by [steady-state.md step A's fix-main-ci reconcile](./steady-state.md#a1-parse-the-return-string) each time a `shipped main-ci-fix` PR for a given workflow signature is followed by the same workflow re-reddening main on the next divert-checks refresh. When `attempts` reaches `main_ci.max_fix_attempts` (default 3, from `shipyard-config.sh`), the [step 4.5a enqueue rule](./setup/04-backlog-divert.md#45-divert-checks-main-ci--pr-pileup) stops enqueueing a `fix-main-ci` divert for that signature, sets `escalated: true`, and surfaces a structured flake escalation banner (recommend quarantine + tracking issue, or human CI-side investigation). Reset for a signature when main goes green on it (the fix worked) — a fresh red on a previously-escalated workflow starts a new attempt cycle only after a human clears the escalation by getting main green. Session-local (not mirrored to the session-state file). Read by [cleanup-summary.md's friction line](./cleanup-summary.md#end-of-session-summary) when any signature escalated.
- **`session_blocked_soft`** ([#300](https://github.com/mattsears18/shipyard/issues/300)): map `{issue_number → ISO-8601 UTC timestamp of the bail}` written by [steady-state.md step A.1's `blocked` handler](./steady-state.md#a1-parse-the-return-string) on soft-classified bails (cannot-reproduce / ambiguous / scope-judgment / PR-already-open), read by [step C's lightweight backlog re-check](./steady-state.md#c-dispatch-a-replacement-if-work-remains--mandatory-action) to skip in-window re-appends. Window = `blocked_agent.soft_retry_minutes` (default 30, from `shipyard-config.sh`); entries clear post-window; map is session-local (not mirrored to the session-state file). Refuse bails apply `needs-human-review` and dependency-wait bails apply no label (gated by the `Blocked by #N` body-reference filter) instead — the former `blocked:agent-hard` label was eliminated in [#521](https://github.com/mattsears18/shipyard/issues/521).
- **`reconciled_agent_ids`** ([#317](https://github.com/mattsears18/shipyard/issues/317)): set of harness `task-id` (a.k.a. `agent_id`) values for which step A has already run a full reconcile this session. Written by [steady-state.md step A's reconcile-once gate](./steady-state.md#a-reconcile-the-return) the moment a return has been parsed; read by the same gate at the **top** of every subsequent turn — when the incoming `task-notification`'s `task-id` is already in the set, the turn is a **phantom re-fire** and the orchestrator MUST silently skip A.0 / A.1 / B / C / D (no token bump, no slot release, no dispatch, no refresh, no invariant line). Session-local (not mirrored to the session-state file) and grows monotonically across the session — no eviction, since the set's only purpose is to recognize a duplicate that has already been fully accounted for. Defends against the observed Claude Code harness pattern where an agent's wind-down chat messages (e.g. "Done.", "Acknowledged.", "Monitor task completed.") arrive as additional `task-notification` events with the same `task-id` after the agent's actual return text was reconciled — see #317 for the repro and the harness-vs-orchestrator-fix tradeoff.
- **`ci_session_counters`** ([#323](https://github.com/mattsears18/shipyard/issues/323)): four-counter map `{ dispatches_skipped_stale_failure, dispatches_deferred_in_progress, drain_rebases_skipped, drain_rebases_dispatched }` tracking the CI-minute discipline gates introduced by the `ci.*` config block. Initialized to all-zero at session start, incremented inline by [steady-state.md step C's pre-dispatch checks](./steady-state.md#c-dispatch-a-replacement-if-work-remains--mandatory-action) (the stale-failure and in-progress-settle gates) and [drain.md's end-of-session drain](./drain.md#end-of-session-drain) (the skip-drain-rebase and max-drain-rebases gates). Session-local (not mirrored to the session-state file). Read by [cleanup-summary.md's `CI cost:` block](./cleanup-summary.md#end-of-session-summary) — surfaced only when at least one counter is non-zero OR at least one `ci.*` config key is non-default, so quiet sessions where the gates were off don't add summary noise. The four counters are intentionally disjoint surfaces (two for steady-state, two for drain) so a reader can attribute each saved CI suite to the specific gate that fired.
- **`primary_leak_counters`** ([#387](https://github.com/mattsears18/shipyard/issues/387)): two-counter map `{ primary_leak_restores, primary_leak_dirty_skips }` tracking how often the [primary-checkout branch-leak guard](./steady-state.md#a06-primary-checkout-branch-leak-guard-fires-every-reconcile-turn-before-a1) fired. The harness `isolation: "worktree"` dispatch path can leak a `do-work/*` branch checkout into the user's primary working tree; the guard (run every reconcile turn AND at [drain entry](./drain.md#end-of-session-drain)) restores a clean primary to the default branch (`primary_leak_restores`) or, when the primary tree is dirty, warns and skips without writing (`primary_leak_dirty_skips`). Initialized to all-zero at session start. Session-local (not mirrored to the session-state file). Read by [cleanup-summary.md's friction line](./cleanup-summary.md#end-of-session-summary) — surfaced only when the combined count is non-zero (silent on quiet sessions, per the `ci_session_counters` precedent).
- **`dispatch_denials`** ([#718](https://github.com/mattsears18/shipyard/issues/718)): list of `{ target: <#N | #M | "main" | "pr-pileup">, mode: "<issue-work|fix-checks-only|fix-rebase|fix-main-ci|fix-failing-prs-batch|investigate>", denied_at: "<iso-8601 UTC>", denial_text: "<verbatim first line of the harness denial>", attempt: 1|2, outcome: "reframed" | "shipped-after-reframe" | "handed-back" }` entries recording every `Agent` dispatch the **Claude Code auto-mode permission classifier refused outright** — the dispatch tool call never happened, so there is no worker, no worktree, no `agent_id`, and no completion notification to reconcile. Written by the [denied-dispatch branch](./dispatch-rules.md#dispatch-denied-by-the-harness-permission-classifier-718) at both dispatch sites ([setup step 7](./setup/07-pool-fill.md#7-initial-pool-fill) and [steady-state step C](./steady-state.md#c-dispatch-a-replacement-if-work-remains--mandatory-action)). **Exactly one** re-dispatch is permitted per target, and only to correct a prompt that *overstated* the work's blast radius (the re-dispatch must make the prompt more **accurate**, never merely more permissive-sounding); iterating wording against the classifier is forbidden, and a second denial hands the target back to the human (`needs-human-review` for issue targets) rather than trying a third time. Session-local (not mirrored to the session-state file). Read by [cleanup-summary.md's `Dispatch denied:` line](./cleanup-summary.md#end-of-session-summary) so a denial is visible rather than silently costing a dispatch slot. Distinct from [#712](https://github.com/mattsears18/shipyard/issues/712)'s *worker-side* Bash denial (handled by the worker's `blocked:` return per `shipyard:worker-preamble` § "After a classifier denial") — this is the orchestrator's own dispatch call being refused.
- **`operator_denials`** ([#746](https://github.com/mattsears18/shipyard/issues/746)): list of `{ kind: "merge-pr" | "close-pr" | "paste-secret" | "toggle-setting" | "reply-comment" | "console-action", target: "<url or #N>", denied_at: "<iso-8601 UTC>", denial_text: "<verbatim first line of the harness denial>", outcome: "reframed" | "shipped-after-reframe" | "handed-back" }` entries recording every **operator action** the Claude Code auto-mode permission classifier refused outright — the mutating call (`gh pr close`/`gh pr merge`/a browser mutation) never executed. Mirrors `dispatch_denials` (#718) but for the operator layer's queue-drain actions rather than `Agent` dispatch calls; the two are recorded separately because the classifier denies them for different reasons (dispatch: widening the *worker's* permissions; operator: acting on an artifact outside the session's own scope). Written by [operate/01-queue-and-authorization.md's denied-operator-action branch](./operate/01-queue-and-authorization.md#operator-action-denied-by-the-harness-permission-classifier-746) whenever a queue-drain attempt is denied. **At most one** re-attempt is permitted per item, and only to cite an explicit user confirmation already on record (see [operate/01-queue-and-authorization.md's batched inherited-PR confirmation](./operate/01-queue-and-authorization.md#scope-of-standing-authorization--session-owned-artifacts-vs-inherited-third-party-prs-746)) — never to iterate wording against the classifier. A second denial (or no confirmation to cite) degrades the item to a `needs-operator` hand-back rather than being retried or silently dropped from `operator_queue`. Session-local (not mirrored to the session-state file). Read by [cleanup-summary.md's `Operator denied:` line](./cleanup-summary.md#end-of-session-summary) so a swallowed operator action is visible rather than silently costing a queue item. Distinct from `dispatch_denials` (the orchestrator's own `Agent` dispatch call being refused) and from [#712](https://github.com/mattsears18/shipyard/issues/712)'s worker-side Bash denial (a `blocked:` return from inside a dispatched worker) — this is the orchestrator's own **operator-phase** action being refused.
- **`version_cursor`** ([#437](https://github.com/mattsears18/shipyard/issues/437)): session-local high-water mark of the highest manifest version slot **claimed by dispatch** — distinct from the open-PR walk the [#339 next-available-version computation](./dispatch-rules.md#dispatch-rules-used-by-step-7-and-step-c) does. The #339 walk only sees versions claimed by **open** PRs, which is blind to siblings dispatched in the same batch (initial pool fill at [setup.md step 7](./setup/07-pool-fill.md#7-initial-pool-fill), or any step C multi-fill): those PRs aren't open yet, so every worker in the batch reads the same floor (main's version) and the per-dispatch walk hands them all the same `next_available_version` — the first C≥2 batch all claims `main+1` and N−1 PRs go DIRTY on the version row. The cursor closes that gap: the version computation seeds from `max(version_cursor, session_prs-walk-result)` and **advances the cursor to the value it hands out**, so a batch of N simultaneous dispatches receives distinct monotonic slots (`main+1`, `main+2`, … `main+N`) even before any PR is open. Initialized to `null`/unset at session start (no claim yet); only consulted when `version_coordination.enabled` and a `manifest_path` is configured (otherwise the cursor is never read or written). Session-local (not mirrored to the session-state file) — the cursor only matters within a live session's batch-dispatch window; a fresh session re-derives the floor from `origin/<default-branch>`'s manifest.
