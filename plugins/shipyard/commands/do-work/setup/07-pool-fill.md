# /shipyard:do-work — Setup phase · initial pool fill

**Setup sub-phase (cluster 5 of 5).** Owns step 7: the initial pool fill that dispatches the first wave of workers, then hands off to [steady-state](../steady-state.md). Router: [`setup.md`](../setup.md). Sidebar: [`dont.md`](../dont.md). Prev: [`06-scope-preflight.md`](./06-scope-preflight.md).

### 7. Initial pool fill

> **Fire one when `concurrency == 1`.** At C=1 the "pool" is a single slot. Skip the parallel dispatch burst and dispatch exactly one worker: apply the dispatch rules from [steady-state dispatch rules](../dispatch-rules.md#dispatch-rules-used-by-step-7-and-step-c) to pick the top candidate, run the just-in-time scope pre-flight for that one candidate (per the C=1 note in [step 6](06-scope-preflight.md#6-initial-scope-pre-flight)), pre-provision its worktree, then issue a single `Workflow` call (no `run_in_background: true` needed — the slot is already available). Return control immediately after dispatch; the steady-state loop handles the rest.

Dispatch up to `--concurrency` workers in parallel — one message with N background `Workflow` calls (`run_in_background: true`) against [`workflows/do-work-dispatch.workflow.js`](../../../workflows/do-work-dispatch.workflow.js), one work unit each, per the [substrate section's five-step call-site walkthrough](../dispatch-rules.md#workflow-substrate-dispatch--the-dispatch-mechanism-for-every-worker-mode-791). **Pre-provision each worker's isolated worktree with `git worktree add` before its call** and pass the absolute path as that unit's `worktreePath` — the workflow runtime has no isolation primitive of its own, and [`enforce-worktree-isolation.sh`](../../../hooks/enforce-worktree-isolation.sh) hard-fails a unit dispatched without one. Each unit also carries the `model` resolved from the merged config's `models.<mode>` key via [`resolve-dispatch-model.sh <mode>`](../../../scripts/resolve-dispatch-model.sh), omitting the field when the resolver returns empty ([#727](https://github.com/mattsears18/shipyard/issues/727) — see the [per-dispatch model-resolution rule](../dispatch-rules.md#dispatch-rules-used-by-step-7-and-step-c)). For each slot, pick the next job using the **dispatch rules** below.

**Pre-allocate distinct manifest versions across the batch when `version_coordination.enabled` ([#437](https://github.com/mattsears18/shipyard/issues/437)).** The N `Workflow` calls fire in one message, so the N sibling PRs do not exist yet — the [steady-state next-available-version `session_prs` walk](../dispatch-rules.md#dispatch-rules-used-by-step-7-and-step-c) is blind to them and would hand every issue-work slot in the batch the same `main+1` slot, reddening N−1 of them on the version row at merge. To prevent the collision, run the next-available-version computation **once per issue-work slot, in sequence**, while composing the batch's prompts: each run reads and advances the session-local `version_cursor` the previous slot's run set, so slot 1 receives `main+1`, slot 2 receives `main+2`, … and each worker's dispatch prompt carries a **distinct** "Next-available version (orchestrator-supplied)" paragraph. The `Workflow` calls still fire simultaneously; only the per-slot version assignment that feeds each prompt is computed serially against the shared cursor. Non-issue-work slots (fix-checks-only, divert workers) don't bump the manifest and are skipped by the computation. When `version_coordination.enabled` is false (or no `manifest_path` is configured), the cursor is never touched and this paragraph is a no-op.

**If the harness permission classifier refuses an in-batch dispatch call** ([#718](https://github.com/mattsears18/shipyard/issues/718)), that dispatch never happened — no agent ran, no completion notification is coming. The other calls in the batch are unaffected. Handle the refused one per [dispatch-rules.md § "Dispatch denied by the harness permission classifier"](../dispatch-rules.md#dispatch-denied-by-the-harness-permission-classifier-718): record it in `dispatch_denials`, **reap the worktree you already pre-provisioned for it** (nothing else will — it has no `.in_flight` slot), take **at most one** *accuracy-correcting* re-dispatch (never iterate wording against the classifier), hand back to the human on a second denial, and fill the slot with the next candidate.

**Per-slot metadata.** Each new `in_flight` slot record MUST include `started_at` (ISO-8601 UTC) alongside `kind` / `target` / `claimed_paths` / the dispatch id / the pre-provisioned `worktree_path`. The write-through runs **after** the `Workflow` call is accepted (the dispatch id doesn't exist before then), so a classifier-denied dispatch leaves no phantom slot. This powers [`/shipyard:status`](../../status.md)'s `ELAPSED` column and stale-worker detection — see the [steady-state per-slot dispatch metadata write-through note](../steady-state.md#c-dispatch-a-replacement-if-work-remains--mandatory-action) for the canonical shape. Same write-through pattern applies to every dispatch site in the session (initial pool fill, step C, divert-queue pop, fix-checks pop, drain-phase fix-rebase dispatch).

Once the pool is full, **return control** — you'll be notified the moment any agent completes. Do not poll. Do not sleep.
