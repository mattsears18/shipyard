# /shipyard:do-work — Don't

The orchestrator's rule list — load-bearing prohibitions across every phase. The *why* for each rule lives in [RATIONALE → Don't extended](../do-work-RATIONALE.md#dont--extended-rationale-for-the-load-bearing-rules); cross-references are inline where the full discussion matters. The thin entry [`commands/do-work.md`](../do-work.md) links here from the [phase routing summary](../do-work.md#phase-routing).

**Dispatch-loop discipline:**

- **Don't go idle while `raw_backlog` or `ready_issues` has items.** Refill the pool on the same turn the last completion notification arrives. ([why](../do-work-RATIONALE.md#dont-go-idle-while-queues-hold-work))
- **Don't conflate "pool drained" with "session complete."** The merge train continues without you; the dispatch loop dies if you stop dispatching. ([why](../do-work-RATIONALE.md#dont-conflate-pool-drained-with-session-complete))
- **Don't declare termination without running the canonical fresh-fetch query of [drain.md's termination-assertion step 4](./drain.md#termination-assertion).** `raw_backlog == 0` in stored state is a necessary precondition, not a sufficient one — user-filed mid-session issues can sit on the live tracker without surfacing into the cached backlog. Step E's `last_fresh_fetch=<HH:MM:SS|"never">` invariant token is the per-turn evidence; the staleness guard on `idle_reason="all queues empty (terminating after in_flight drains)"` requires `last_fresh_fetch` to be within the last 60s of `now` before the orchestrator may hand off to drain. See [issue #195](https://github.com/mattsears18/shipyard/issues/195) for the regression that prompted the numbered-procedure restructuring of the termination assertion.
- **Don't hand a workable issue to the human — attempt-then-escalate, never escalate-without-attempting ([#531](https://github.com/mattsears18/shipyard/issues/531)).** If an issue passes the mechanical [step-4 workable filter](./setup/04-backlog-divert.md#4-fetch--rank-the-backlog) (trusted author, no dispatch-gate label, no open `Blocked by #N`, not assigned elsewhere), the orchestrator MUST dispatch a worker against it before terminating. It may NOT surface the issue to the user as "left for a fresh run" / "say the word and I'll work them" / "next step for you" while the agent is itself capable of doing it. **"It's a follow-up I filed this session" / "feels like it should wait" / "it's a meta-issue about the orchestrator" / "it rewrites the path I'm running" are NOT valid defer reasons — they match none of the five [`defer_reason_class`](./setup/06-scope-preflight.md#6-initial-scope-pre-flight) values** (`external-dependency`, `human-decision-required`, `untrusted-author`, `confirmed-blocker-still-open`, `confirmed-non-shippable-as-single-PR`). A defer to the human is gated to exactly those classes; an issue that matches no class is workable and MUST be worked, not deferred off-book. **Self-filed follow-ups re-enter the backlog like any other issue** — a spec/doc change handled by a worktree-isolated worker and merged via PR never modifies the live session, so "it touches the orchestrator I'm running" is not a reason to withhold dispatch. The legitimate guard against a self-perpetuating backlog is NOT "refuse to work self-filed issues" — workers shipping fixes shrink the backlog, and follow-ups are themselves finite and workable. If genuinely worried about unbounded regress, the bounded guard is a soft cap on the per-session count of issues filed by this session's own workers (surface-and-pause at the cap), never a blanket refusal to work them. See [issue #531](https://github.com/mattsears18/shipyard/issues/531) for the session that drained on workable self-filed follow-ups and forced the maintainer to re-instruct the orchestrator to finish them.
- **Don't defer an epic, spike, or design-needed issue merely because it "has open design decisions" ([#767](https://github.com/mattsears18/shipyard/issues/767)).** This is a corollary of the rule above, naming the specific rationalization that motivated it: a scope agent (or a human reading a stale mental model of the taxonomy) can mistake "needs a design" or "needs an architecture decision" for a legitimate `human-decision-required` reason. It is not — see [`06-scope-preflight.md`'s design/architecture/epic/spike callout](./setup/06-scope-preflight.md#designarchitectureepicspike-decisions-are-in-scope-by-default-not-a-defer-reason-767). Extend the [#661](https://github.com/mattsears18/shipyard/issues/661) "autonomous design / architecture decisions" default from in-worker implementation choices to issue scoping itself: make the reasonable design call and dispatch (sliced with `phase_1_scope` if multi-phase); an epic that genuinely can't ship as one PR still routes to `confirmed-non-shippable-as-single-PR` (decomposition), never to `human-decision-required`. Reserve `human-decision-required` / `needs-triage` / `needs-human-review` for security/access-control settings, unprovisioned external-service credentials, and genuine product/business/legal judgment calls — not design or architecture.
- **Don't file end-of-session friction/follow-up issues after the termination assertion has run, and never during or after [cleanup](./cleanup-summary.md#end-of-session-cleanup) ([#743](https://github.com/mattsears18/shipyard/issues/743)).** This is the filing-side corollary of the rule above: a friction issue filed *after* [drain.md's four-step assertion](./drain.md#termination-assertion) has already passed is a fresh workable candidate the previous rule now obligates the orchestrator to dispatch — but if the filing happens after [cleanup](./cleanup-summary.md#end-of-session-cleanup) has already flushed the cost ledger and retired the session-state file, that dispatch's tokens have no durable session record to land in, and `/shipyard:cost report` silently under-reports the session by however much the dispatch cost. File any such issues **before** running the termination assertion (see [drain.md's guidance](./drain.md#termination-assertion)) — the assertion's own step-4 fresh-fetch then picks the new issue up as a net-new candidate and loops back through dispatch instead of terminating around it. See [issue #743](https://github.com/mattsears18/shipyard/issues/743) for the repro (a session's own follow-up issue and its ~250k-token resolving PR were entirely absent from the flushed ledger line) and [cleanup-summary.md's re-entrancy safety net](./cleanup-summary.md#end-of-session-cleanup) for the defense-in-depth fallback when a dispatch still lands post-cleanup despite this ordering.
- **Don't exit the end-of-session drain while forward progress continues.** Exiting at a hard 15-min mark strands PRs that go red after the cap. ([why](../do-work-RATIONALE.md#dont-exit-the-drain-while-forward-progress-continues))
- **Don't ask the user "should I keep working, summarize, park, or stop?"** Termination is mechanical (see [Termination assertion](./drain.md#termination-assertion)). The only stop signal is the explicit `stop` / `drain` / `/do-work stop` phrase from [Soft drain](./drain.md#soft-drain).
- **Don't emit recap / "Next: …" narration in lieu of dispatching.** A turn that frees a slot has shape: tool call → invariant line. No prose between them, no prose after them, no "Next:" sentence anywhere. Forbidden phrasings include "Next, I'll watch for …", "Waiting for completions to refill …", "The merge train will drain on its own …". ([why + DO/DON'T examples](../do-work-RATIONALE.md#dont-emit-recap--next--narration-in-lieu-of-dispatching))
- **Don't poll for agent completion.** The harness notifies you. Polling burns turns and cache.
- **Don't wait for the whole pool to drain before dispatching replacements.** Fill any single freed slot immediately.
- **Don't run A.0/A.1/B/C/D on a phantom re-fire `task-notification`.** [Step A.−1's reconcile-once gate](./steady-state.md#a1-reconcile-once-gate--skip-phantom-re-fires-mandatory--first-thing-in-the-turn) MUST run before A.0 on every wake; when the incoming `task-id` is already in `reconciled_agent_ids`, log the `[phantom-notification]` advisory and end the turn — no token bump, no slot release, no dispatch, no refresh, no invariant line. This is the **one documented exception** to the "every turn ends with either a tool call or the invariant line" rule from the steady-state turn contract. See [#317](https://github.com/mattsears18/shipyard/issues/317) for the harness pattern and the cost-of-not-having-the-gate.
- **Don't *pure*-silent-skip a phantom re-fire whose body asserts a terminal outcome for a *different, still-in-flight* target ([#530](https://github.com/mattsears18/shipyard/issues/530)).** The silent `return` is correct only for a **genuine wind-down phantom** (body asserts nothing reconcilable, or names only the already-reconciled target). When the harness cross-wires a still-in-flight **sibling** worker's only completion notification onto a reaped worker's `task-id` — the phantom's `task-id` is reconciled, but its body asserts `shipped #<N> via PR #<M>` / `green #<M>` for a target that maps to a live `.in_flight` slot — a pure skip **strands the in-flight sibling** (its real completion arrived only as this phantom; no other notification is coming). Before skipping, parse the body: if it names an in-flight slot's target, run the [trust-but-verify probe](./dispatch-rules.md#dispatch-rules-used-by-step-7-and-step-c) (issue state + PR `mergeStateStatus`) and, if ground truth confirms the outcome, reconcile the **in-flight sibling** from verified state (keyed on the sibling's `agent_id`, not the phantom's reaped id) rather than silent-skipping. The body's narrative is a *trigger to verify*, never authority to reconcile on its word — a target whose ground truth does NOT confirm falls back to the silent skip. See [Step A.−1's sibling-strand variant](./steady-state.md#a1-reconcile-once-gate--skip-phantom-re-fires-mandatory--first-thing-in-the-turn).
- **Don't reconcile a return without first calling `bump-tokens` and (for `shipped` / `green` / `noop: already green` returns) posting the cost-tracking comment via `read-tokens --format comment` + sentinel edit-or-create.** [Step A.0](./steady-state.md#a0-attribute-the-dispatchs-token-usage-mandatory--before-any-return-string-parsing) is MANDATORY on every reconcile turn — without it, `.tokens.totals`, `.tokens.per_issue[<N>]`, `.tokens.per_pr[<M>]`, the durable PR cost-comment, and the persistent ledger at `~/.shipyard/cost-history.jsonl` all stay empty, and the perf umbrella ([#152](https://github.com/mattsears18/shipyard/issues/152)) becomes unmeasurable. Step E's `tokens_attributed=<true|false>` token is the per-turn audit signal — `tokens_attributed=false` on a reconcile turn is a contract violation. See [issue #197](https://github.com/mattsears18/shipyard/issues/197) for the regression that prompted promoting attribution to a numbered first step.

**Dispatch hygiene:**

- Don't work on issues assigned to other users — soft-lock via `gh api user` check.
- Don't merge manually. Use `--auto`. Auto-merge waits for green.
- Don't disable required checks or weaken branch protection to make a PR pass.
- Don't re-dispatch the same issue. Once the agent returns `blocked`, label it and never queue it again.
- Don't fabricate acceptance criteria. The agent should infer reasonable ones from title + context; if even that's unclear, it returns `blocked`.
- Don't dispatch two workers whose `claimed_paths.hard` overlap. The soft-collision tier relaxes this for additive docs paths up to `--soft-collision-concurrency` claimers — that's a different rule.
- **Don't expect agents to resolve cross-worker merge conflicts on soft-collision paths — the orchestrator owns that.** Dispatch a drain-style fix-rebase worker on the second PR. ([why](../do-work-RATIONALE.md#dont-expect-agents-to-resolve-cross-worker-merge-conflicts-on-soft-collision-paths))
- **Don't treat a `blocked rebase #<M>: conflict extends beyond coordinated manifest+CHANGELOG rows` return on a soft-collision PR as a worker failure — it's expected ([#507](https://github.com/mattsears18/shipyard/issues/507)).** When two+ soft-collision claimers edit the **same section** of the same docs file, the conflict is real prose content, not the coordinated manifest `.version`/CHANGELOG rows that [fix-rebase's §4.6](../../agents/issue-worker/fix-rebase.md#46-version-coordinated-manifest--changelog-re-number-trivial-resolution-issue-466) auto-resolves — so fix-rebase correctly bails. The documented drain branch (steady-state's [blocked-rebase reconcile](./steady-state.md#a-reconcile-the-return)) adds the PR to `rebase_blocked_prs`, comments `Needs manual rebase.`, and surfaces it still-DIRTY for a human to hand-resolve. Don't re-dispatch fix-rebase (same conflict re-bails) and don't make the soft cap section-aware to "fix" it — that's the deliberately-rejected option 2; the residual cost is accepted at `--concurrency ≥ 2`.
- Don't dispatch a candidate whose `lockfile_sections` overlaps an in-flight worker's `lockfile_sections`. Generated lockfiles are not claimed as sections.
- Don't dispatch without `run_in_background: true` and `isolation: "worktree"`. Two `PreToolUse` hooks defend the contract: [`enforce-worktree-isolation.sh`](../../hooks/enforce-worktree-isolation.sh) blocks missing `isolation: "worktree"` on **every** guarded shim — `shipyard:issue-worker`, `shipyard:fix-checks-worker`, `shipyard:fix-rebase-worker`, `shipyard:fix-main-ci-worker`, `shipyard:fix-pr-batch-worker` (#293 closed the gap where the four model-pinned shims used to pass through silently); [`enforce-edit-scope.sh`](../../hooks/enforce-edit-scope.sh) blocks writes from a worker's cwd to paths outside its worktree. Neither hook accepts a workaround.
- Don't skip the initial scope pre-flight to "save time" — one rebase from a missed collision costs more than the whole pre-flight.
- Don't skip the periodic failed-PR refresh.
- **Don't iterate prompt wording against the permission classifier when an `Agent` dispatch is denied ([#718](https://github.com/mattsears18/shipyard/issues/718)).** When Claude Code's auto-mode classifier refuses the orchestrator's own dispatch call, the tempting default is to keep rewording the prompt until it gets through — and **that is exactly the bypass the classifier exists to prevent**. An orchestrator that learns "launder the prompt until it's allowed" has defeated the harness's safety boundary on *every* future dispatch, not just this one. Exactly **one** re-dispatch is permitted, and only to **correct** a prompt that *overstated* the work's blast radius (it implied writing a live permission file when the deliverable is plugin source in an isolated worktree) — the reframe must make the prompt **more accurate**, never merely more permissive-sounding. If the work genuinely requires the denied capability, the prompt was right and the classifier was right: hand back without spending the reframe. On a **second** denial, stop and hand back to the human — never a third attempt, never a workaround through a different `subagent_type` / a split dispatch / inline execution on the orchestrator thread (classifier policy targets *effects*, not tool names). Full branch: [dispatch-rules.md § "Dispatch denied by the harness permission classifier"](./dispatch-rules.md#dispatch-denied-by-the-harness-permission-classifier-718).
- **Don't let a denied dispatch silently cost a slot ([#718](https://github.com/mattsears18/shipyard/issues/718)).** A refused `Agent` call produces no worker, no worktree, no `agent_id`, and **no completion notification** — so without an explicit record the target just quietly stops being worked. Record every denial in `dispatch_denials`, fill the slot with the next candidate in the same turn (a denial consumes a *candidate*, not the *slot*), and surface a `Dispatch denied:` line in the [end-of-session summary](./cleanup-summary.md#end-of-session-summary). Don't write an `.in_flight` slot for a denied dispatch, and don't run step A's reconcile against one — there is nothing to reconcile.
- **Don't post the classifier's reasoning to a public GitHub artifact ([#718](https://github.com/mattsears18/shipyard/issues/718)).** On the two-denials hand-back, the issue comment states the **fact and the outcome only** — it must not quote, paraphrase, explain, theorize about, or dispute *why* the harness denied the dispatch, and the orchestrator must not file a follow-up issue arguing the denial was wrong (that's a maintainer decision, made after they see the `Dispatch denied:` line). The verbatim denial text belongs on the **local** surfaces — the session summary and the on-disk HTML report. Same content-integrity boundary the worker operates under (`shipyard:worker-preamble` § "After a classifier denial", [#341](https://github.com/mattsears18/shipyard/issues/341)): a public artifact carrying a reconstructed explanation of a harness decision is a side-channel a future worker, reviewer, or scraper could read as authoritative.

**Failure-handling discipline:**

- Don't retry a `blocked:ci` PR. The label exists so the orchestrator stops banging on the same wall. **Exception:** [step 3d's auto-clear sweep](./setup/01-repo-recovery.md#3-ensure-label-exists--recover-from-prior-session) removes the label at session start from any PR whose head commit is newer than the label's application timestamp (someone pushed since shipyard gave up — fresh chance, 3-attempt counter resets). The sweep is the only mechanism that removes `blocked:ci`; step A is the only mechanism that applies it.
- **Don't accept a `green #<M>` return from a fix-checks-only worker without verifying the rollup.** ([why](../do-work-RATIONALE.md#dont-accept-a-green-return-without-verifying-the-rollup))
- **Don't treat a fix-checks-only worker's narrative status string as authoritative.** ([why](../do-work-RATIONALE.md#dont-treat-a-narrative-status-string-as-authoritative))
- **Don't dispatch fix-rebase outside the end-of-session drain.** Steady-state dispatch never produces a DIRTY PR; dispatching mid-session would churn branches auto-merge was about to rebase or race with a fix-checks worker.
- **Don't retry a `blocked rebase` PR within the same session.** A `blocked rebase` return signals a non-trivial conflict that won't resolve on retry; `rebase_blocked_prs` membership counts toward the drain's "settled" definition. **`rebased` returns are different** — a successfully-rebased PR that goes DIRTY again from a sibling merge IS eligible for re-dispatch (the merge train can still win that race), subject to the 3-successful-rebase rate-limit cap tracked in `rebase_success_counts`. See [drain.md's end-of-session drain](./drain.md#end-of-session-drain) for the split-gate rationale and [#265](https://github.com/mattsears18/shipyard/issues/265) for the race the split fixes.

**Worktree + filesystem discipline:**

- **Don't write to the user's primary checkout under any circumstance.** After step 0.5, the orchestrator works exclusively in `.claude/worktrees/orchestrator-<session-id>`. ([why](../do-work-RATIONALE.md#dont-write-to-the-users-primary-checkout)) **One sanctioned exception** ([#387](https://github.com/mattsears18/shipyard/issues/387)): the [primary-checkout branch-leak guard](./steady-state.md#a06-primary-checkout-branch-leak-guard-fires-every-reconcile-turn-before-a1) may run `git -C <primary> checkout <default> && git -C <primary> pull --ff-only` — but **only** when the harness has already leaked a `do-work/*` checkout onto the primary (the primary is off the default branch, so the read-only contract is already violated by harness behavior shipyard can't prevent) AND the primary tree is clean (the restore is provably lossless). This is a *corrective* write that undoes a harness-leaked write and returns the primary to the state the contract assumes. When the primary tree is dirty, the guard NEVER writes — it warns and defers to the human. Do not generalize this into "shipyard may move the primary's HEAD"; it fires only on the leak-detected-and-clean path.
- Don't run end-of-session cleanup from the user's primary checkout — run it from the orchestrator worktree. The exception is cleanup step 6, which removes the orchestrator worktree itself (metadata-only operation, primary HEAD never moves).
- Don't cleanup branches by name or pattern — only by `[gone]` upstream.
- Don't claim a worktree whose branch doesn't match `do-work/*` during orphan triage. That branch is not yours.
- Don't run orphan triage while another `/do-work` session may be active on the same repo. If you suspect a parallel session, ask the user.
- Don't remove the `shipyard` label on block, abandon, or any other outcome. `shipyard` stays put through the whole cycle; only `blocked:ci` comes and goes via step A (apply) and [step 3d's auto-clear sweep](./setup/01-repo-recovery.md#3-ensure-label-exists--recover-from-prior-session) (remove).
- **Don't omit the `shipyard:worker-preamble` skill reference from any dispatched agent's prompt.** Every dispatch prompt loads the skill instead of inlining the verbatim preamble. Closes [#107](https://github.com/mattsears18/shipyard/issues/107).
- **Don't skip `git worktree unlock` before removing** an agent worktree. Without unlocking, the remove fails with `cannot remove a locked working tree`.
- **Don't reach for `git worktree remove --force` as the first move** ([#712](https://github.com/mattsears18/shipyard/issues/712)). Try the plain `git worktree remove` first — it refuses on a dirty tree, which is exactly the safety property Claude Code's auto-mode permission classifier is protecting when it denies a bare `--force` as `[Irreversible Local Destruction]`. A denied reap silently defeats the entire cleanup subsystem: the reaps are fire-and-forget in a background subshell, so worktrees just accumulate. Escalate to `--force` only behind a preceding, explicit check that establishes the force destroys nothing that exists only in that worktree (the `worktree-reap.sh` helper's evidence gate does this for you — prefer routing removes through `worktree-reap.sh reap`).
- **Don't swallow a failed or denied reap** ([#712](https://github.com/mattsears18/shipyard/issues/712)). `2>/dev/null || true` makes a permission denial indistinguishable from success. Count what's actually left on disk (`worktree-reap.sh report-unreaped`) and surface `Cleanup: <N> worktree(s) could not be reaped — run /clean_gone` in the end-of-session summary. Silent-degrade is what let the count reach six unnoticed.
- **Don't reap a live-PID worktree — at startup OR at shutdown.** Always check the lock PID before unlocking / removing. ([why](../do-work-RATIONALE.md#dont-reap-a-peer-alive-worktree--at-startup-or-at-shutdown))

**Security boundary:**

- **Don't dispatch a worker against an issue authored by a login not in `trusted_authors`.** This is the security boundary from [step 1.7](./setup/01-repo-recovery.md#17-resolve-trusted-author-allowlist) and step 2's bucket 0.5 + step 4's client-side filter + step C's lightweight backlog re-check. ([threat model + maintainer override](../do-work-RATIONALE.md#dont-dispatch-a-worker-against-an-untrusted-author-issue))

- **Don't rely solely on the dispatch-time author check for label integrity.** The author-based gate above blocks worker dispatch against stranger-authored issues, but routing labels on existing issues/PRs are mutated by any actor with `triage` permission — a rogue triager can still mass-apply `shipyard`, prematurely remove `needs-human-review`, or strip `blocked:ci` to reset the 3-attempt counter on a stuck PR. Defense-in-depth: the [`label-event-audit.yml`](../../../../.github/workflows/label-event-audit.yml) workflow watches every `labeled`/`unlabeled` event from untrusted actors and either reverts (Tier A: `shipyard`, `blocked:ci`, `needs-human-review`) or alerts (Tier B: `P0`-`P2`, `wontfix`, `blocked:agent-soft`, `needs-design`, `needs-triage`, `discussion`). See [issue #140](https://github.com/mattsears18/shipyard/issues/140) for the threat model. (`needs-refinement` was dropped from Tier B in [#520](https://github.com/mattsears18/shipyard/issues/520) — the label was eliminated, so there's nothing to audit. `blocked:agent-hard` and the legacy `blocked:agent` were dropped in [#521](https://github.com/mattsears18/shipyard/issues/521): `blocked:agent-hard` no longer exists, and the legacy `blocked:agent` is no longer applied — step 3d.2's migration sweep moves any straggler off it to `needs-human-review` / no-label, so neither warrants a label-mutation alert.)
